# LunaCode Claude 风格 TUI 视觉升级 Spec

## 背景

LunaCode 当前已经具备滚动式终端对话、流式回复、斜杠命令补全、工具运行提示、会话与记忆状态展示等基础交互能力，但界面主要以无样式文本和状态标签输出，启动信息、用户消息、模型回复、工具过程和异常提示之间的视觉层级不够清晰。运行期间的状态变化依赖新增文本行表达，缺少持续更新的加载反馈，状态信息也容易与输入提示混在一起。

本阶段在保留现有滚动式对话体验的前提下，参考 Claude Code 的紧凑终端交互方式，对启动区域、消息标识、加载动画、工具记录、颜色语义和状态显示进行统一升级。视觉主题采用 Luna 月夜风格，不追求像素级复制 Claude Code，也不改造成固定面板式 Dashboard。

## 目标

- 建立清晰、一致且具有 LunaCode 品牌特征的终端视觉语言。
- 保留滚动式对话记录和现有键盘交互，不引入全屏面板布局。
- 为等待模型、模型处理、工具执行、上下文压缩和后台任务提供连续、可取消的动态反馈。
- 将工具调用压缩成易扫描的单行生命周期记录，减少工具过程对主要对话的干扰。
- 优化输入区附近的状态显示，默认只展示决策所需的核心信息，详细信息仍由现有状态命令提供。
- 在现代终端中提供 Unicode 和 ANSI 颜色效果，并在终端能力不足时安全退化。
- 保证动画刷新不会破坏流式回复、当前输入、光标位置、命令补全和终端滚动记录。

## 功能需求

- F1: TUI 保持滚动式对话布局。已经完成的用户消息、模型回复、工具记录和系统提示作为终端历史保留，输入区域位于最新输出之后，不改为固定分栏或全屏 Dashboard。
- F2: 启动后显示紧凑品牌卡片，至少包含 LunaCode 品牌标识、当前版本或构建标识、当前模型、当前工作目录和一条快捷操作提示；品牌卡片不得占用过多对话空间。
- F3: 用户消息、模型回复、工具活动、普通提示、成功、警告、错误和等待确认必须具有稳定且可区分的文本标识，不能继续依赖形如 `[complete]`、`[streaming]` 的内部状态标签作为主要视觉表达。
- F4: 默认主题采用 Luna 月夜配色：品牌和主要交互使用紫色与蓝色，工具活动使用青色，成功使用绿色，警告使用黄色，错误使用红色，次要元数据使用弱化灰色。
- F5: 颜色不能是状态识别的唯一依据。成功、失败、警告、等待和运行中状态还必须通过符号或明确文本区分，以便在纯文本模式下保持可读。
- F6: 系统在等待模型首个输出、模型处理、工具执行、上下文压缩和后台任务处理期间显示统一风格的动态加载指示器，并同时展示当前动作摘要。
- F7: 动态加载指示器必须在没有新模型事件到达时仍能按稳定节奏更新，并显示从当前动作开始计算的已耗时；动作完成、失败或取消后必须立即停止。
- F8: 动态状态必须在同一临时行原位更新，不得为每一帧追加新行。状态切换或结束后，临时行应被正确清除或转化为最终记录，不留下重复 Spinner、残余字符或 ANSI 控制码。
- F9: 运行中的临时状态行至少展示加载指示器、当前动作、已耗时和取消提示；用户仍可通过既有 Esc 或取消命令中止当前工作。
- F10: 工具调用采用紧凑生命周期记录。执行中显示加载指示器、工具名和可安全展示的目标摘要；成功后同一记录转化为成功符号、工具名、目标摘要和耗时；失败后转化为失败符号和简短错误摘要。
- F11: 工具记录默认不展开完整参数、完整输出或长错误堆栈，避免淹没对话；模型仍能收到完整工具结果，用户可通过最终回复、错误摘要或已有诊断方式了解详情。
- F12: 空闲状态下，输入区附近展示紧凑上下文信息，默认包含当前模型、Agent 模式、权限模式和会话标识；缺失字段应被省略，不使用无意义占位符堆积界面。
- F13: Token 用量、Memory 状态、Provider 等详细运行信息不在紧凑状态中常驻展示，继续通过现有状态命令查看；发生与这些信息直接相关的异常时，允许在临时状态或警告记录中展示必要摘要。
- F14: 等待用户回答、等待权限确认、已取消、警告和错误状态必须使用独立且明确的最终提示，不得继续显示运行中动画；提示内容应保留会话或权限等必要上下文。
- F15: 动画更新、状态变化和流式文本到达期间，用户已经输入的内容、输入光标位置和命令补全状态不得丢失、重复或错位。
- F16: 命令补全候选使用与主题一致的视觉层级，候选命令和别名归属易于区分；显示或关闭候选列表后，输入行必须正确恢复。
- F17: 清屏后重新绘制必要的品牌或紧凑上下文与输入区域，并重置旧消息、旧动画和旧状态的渲染缓存，避免清屏后出现残留或漏绘。
- F18: 在支持 ANSI 和 Unicode 的交互式现代终端中启用完整颜色、符号和动画；检测到 `NO_COLOR`、非交互输出或能力不足时，自动退化为无颜色的 ASCII 文本与 ASCII 动画。
- F19: 终端宽度不足或运行中发生尺寸变化时，品牌卡片、状态行、工具摘要和补全候选不得破坏输入行；过长的次要信息应安全截断或换行，核心状态和用户输入优先保留。
- F20: 退出、取消、异常或正常关闭时必须恢复终端属性并重置颜色样式，不能影响用户返回后的 Shell 提示符。

## 非功能需求

- N1: 动画刷新不得使用忙等待；空闲和运行时的额外 CPU 消耗应保持低水平，且不得阻塞模型流读取、工具执行或键盘输入。
- N2: 来自流式响应、后台任务、动画计时和用户输入的并发刷新必须保持线程安全，不能出现交错 ANSI 序列、重复内容或终端状态损坏。
- N3: 加载动画开始、动作切换、完成、失败和取消的视觉反馈应及时；正常终端中不应出现用户可感知的长时间冻结。
- N4: 所有颜色输出必须在局部正确重置，任何消息、状态行或异常路径都不能把样式泄漏到后续输出。
- N5: 视觉升级不得改变对话内容、Agent Loop 决策、工具调用参数、权限判断、会话持久化或 Provider 协议行为。
- N6: 现有输入编辑、方向键、Home/End、删除、Tab 补全、Enter 提交、Esc 退出或取消、斜杠命令和流式回复能力不得回退。
- N7: 动态行为应具备可重复的自动化验证方式，测试不能依赖不可控的真实时间等待或特定终端主题才能判断结果。
- N8: Unicode 符号和中英文混排的显示宽度必须被正确处理，不能因视觉列宽与字符数量不一致而破坏光标定位。
- N9: Windows Terminal、常见 PowerShell 终端以及常见 Linux/macOS ANSI 终端应获得一致、可理解的核心交互；终端差异只允许影响装饰效果，不得影响操作能力。
- N10: 用户可见文案应简洁一致，优先描述正在做什么和结果如何，避免暴露内部枚举名或实现状态名。

## 不做的事

- 不实现固定面板、分栏、鼠标交互或独立滚动区域的全屏 Dashboard。
- 不实现与 Claude Code 完全一致的品牌、Logo、文案或像素级复制，只借鉴其紧凑交互和信息层级。
- 不在本阶段提供可配置主题编辑器或任意颜色配置；只提供 Luna 月夜默认主题和无颜色降级。
- 不实现完整 Markdown 排版、语法高亮、代码块分页或富文本渲染。
- 不默认展开完整工具输入、工具输出、调用栈或调试日志。
- 不新增 Agent、工具、权限、记忆、上下文压缩或后台任务的业务行为；只消费已有状态并改善用户可见反馈。
- 不改变现有快捷键语义，不新增鼠标按钮、复制按钮或交互式折叠控件。
- 不保证所有历史终端模拟器获得相同装饰效果，但必须提供可操作的纯文本降级。

## 验收标准

- AC1: 启动 LunaCode 后，用户看到紧凑的 LunaCode 品牌卡片，其中包含品牌标识、版本或构建标识、当前模型、工作目录和快捷提示，随后可正常输入消息。（对应 F1、F2）
- AC2: 提交一条普通消息后，用户消息和流式模型回复使用清晰的角色标识与 Luna 月夜配色展示，不再主要显示 `[complete]` 或 `[streaming]` 内部标签。（对应 F3、F4）
- AC3: 在禁用颜色的环境运行同一场景时，所有角色和成功、警告、错误状态仍可通过符号或文本区分，输出中不包含可见 ANSI 乱码。（对应 F5、F18）
- AC4: 模型尚未返回首个文本时，界面持续显示会变化的加载指示器、当前动作、递增耗时和取消提示；首个文本到达或动作结束后动画及时停止。（对应 F6、F7、F9）
- AC5: 工具、上下文压缩和后台任务运行时分别显示符合当前动作的加载摘要，而不是统一显示含义模糊的 `responding` 或 `tool_running` 内部状态名。（对应 F6、F10）
- AC6: 观察多个动画帧时，终端历史中只有一条原位更新的临时状态，不产生逐帧新增行；完成后没有残余字符、Spinner 或颜色控制状态。（对应 F8）
- AC7: 工具成功执行后，执行中记录转化为包含成功符号、工具名、目标摘要和耗时的紧凑记录，并且不打印完整工具结果。（对应 F10、F11）
- AC8: 工具执行失败后，动画停止并留下包含失败符号和可理解错误摘要的记录，随后用户仍可继续输入或取消任务。（对应 F10、F14）
- AC9: 空闲输入区附近只常驻显示模型、Agent 模式、权限模式和会话标识；Token、Memory 和 Provider 详情可通过现有状态命令查看。（对应 F12、F13）
- AC10: 等待用户回答或权限确认时，界面显示明确问题或权限提示且不继续运行 Spinner；取消、警告和错误也各自显示清晰的最终状态。（对应 F14）
- AC11: 在动画持续更新和模型流式输出时输入并编辑一段文本，输入内容、光标位置和最终提交内容均保持正确。（对应 F15）
- AC12: 输入斜杠命令并触发多个补全候选时，候选列表层级清晰；关闭候选或选择补全后，输入行恢复且字符和光标位置正确。（对应 F16）
- AC13: 执行清屏后，旧对话、旧动画和旧状态不再出现，必要上下文和输入区域被正确重绘，后续消息仍可正常显示。（对应 F17）
- AC14: 在设置 `NO_COLOR`、非交互输出或模拟不支持 Unicode 的终端条件下，界面自动使用 ASCII 符号和纯文本动画，核心操作保持可用。（对应 F18）
- AC15: 在窄终端和运行中调整终端宽度时，长模型名、工具目标、状态摘要和候选列表不会覆盖或破坏用户输入行。（对应 F19）
- AC16: 正常退出、Esc 退出、运行中取消和异常退出测试后，Shell 的颜色、光标和输入模式恢复正常。（对应 F20）
- AC17: 动画运行期间模型流式文本、键盘输入和工具状态能够同时推进，没有明显卡顿、忙等待、交错控制码或重复输出。（对应 N1、N2、N3）
- AC18: 运行现有 TUI、命令补全、状态上下文、Agent Loop、权限和会话相关测试时全部通过，证明视觉升级未改变核心业务行为。（对应 N5、N6）
- AC19: 自动化测试能够在不依赖真实模型和长时间睡眠的条件下验证颜色降级、动画帧、耗时、状态切换、宽度截断和终端恢复。（对应 N7、N8、N9）
- AC20: 在 tmux 中启动 LunaCode，提交一个需要读取多个文件并总结的真实请求，可观察到品牌卡片、模型加载动画、紧凑工具记录、流式回复、紧凑状态和最终恢复输入的完整流程。（对应全部核心功能需求）
