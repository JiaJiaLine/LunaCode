# LunaCode 上下文管理 Spec

## 背景

LunaCode 当前已经具备终端 TUI 对话、工具系统、多轮 Agent Loop、权限系统、MCP 工具接入、结构化 System Prompt、Provider 适配和基础对话历史管理。随着 Agent 长时间工作，对话历史、工具调用结果、文件内容、报错日志和调试过程会持续累积，最终逼近模型上下文窗口上限。

上下文膨胀的主要来源不是用户原始消息，而是工具结果、文件读取内容、命令输出和多轮调试痕迹。如果 LunaCode 只把完整历史不断追加给模型，长任务会在上下文接近上限时变得不稳定：轻则请求失败，重则模型看不到关键上下文或被旧任务噪声干扰。

本阶段目标是给 LunaCode 增加会话内上下文管理能力，通过两层压缩策略保证有限 Token 预算内的长时间工作能力。第一层是轻量预防，把过大的工具结果外置到会话临时文件，对话里只留下预览和可重新读取的路径；第二层是重量兜底，在整体历史逼近窗口上限或用户主动输入 `/compact` 时，调用当前模型生成结构化摘要，压缩较早历史，保留近期原文，并附加必要的恢复上下文。

本阶段只服务当前会话。跨会话记忆、长期项目记忆、用户级全局记忆和记忆系统留到后续章节实现。

## 目标

- LunaCode 能在每次模型请求前估算上下文体积，并按配置的上下文窗口和压缩阈值决定是否压缩。
- LunaCode 能优先压缩工具结果，把大工具结果外置到项目内会话临时目录，并在对话中保留预览和文件路径。
- LunaCode 能在自动触发、强制触发或用户手动输入 `/compact` 时，使用同一套重量压缩逻辑生成结构化摘要。
- LunaCode 能保留近期原文消息，且不切断工具调用和工具结果之间的配对关系。
- LunaCode 能在压缩摘要后附加完整会话记录路径、最近访问文件快照、已使用 Skill 定义和完整工具声明所需上下文，防止模型根据摘要脑补细节。
- LunaCode 能在摘要失败时熔断普通自动压缩，并在强制压缩失败时停止当前请求并提示用户手动处理。
- LunaCode 能用近似 Token 估算完成本阶段目标，不引入精确 tokenizer 或机器学习式摘要策略优化。

## 功能需求

- F1: LunaCode 提供可配置的上下文窗口上限。上下文窗口由配置项声明，不根据模型名猜测。
- F2: LunaCode 提供上下文压缩相关配置，并支持合理默认值。默认单个工具结果外置阈值为 50,000 字符，默认单条工具结果消息聚合限制为 200,000 字符，默认摘要输出预留为 20,000 token，默认自动压缩安全余量为 13,000 token，默认强制压缩线为自动压缩阈值再增加 10,000 token。
- F3: LunaCode 按固定公式计算压缩阈值：有效窗口等于上下文窗口减去摘要输出预留；自动压缩阈值等于有效窗口减去自动压缩安全余量；强制压缩阈值等于自动压缩阈值加 10,000 token。
- F4: LunaCode 在每次模型请求前先执行轻量预防，再检查重量压缩。轻量预防只处理单条消息中的工具结果大小，重量压缩处理累积历史长度。
- F5: 轻量预防优先处理工具结果。单个工具结果超过阈值时，LunaCode 将完整工具结果写入当前会话的项目内临时目录，对话历史中只保留工具名、结果摘要、预览、完整文件路径和重新读取提示。
- F6: 单条工具结果消息内多个工具结果的字符数合计超过聚合限制时，LunaCode 按工具结果大小从大到小依次外置，直到该消息内剩余工具结果总字符数降到预算以内。
- F7: 外置工具结果文件只保证当前会话可用。文件路径位于项目内会话临时目录，模型可以通过普通文件读取能力重新读取；LunaCode 不承诺重启后仍能恢复这些文件。
- F8: LunaCode 在压缩前写出当前会话完整记录文件。记录内容包含完整消息历史、外置工具结果索引和必要元数据；摘要消息中必须包含完整会话记录路径，提示模型需要压缩前细节时读取会话记录或重新读取项目文件。
- F9: 自动压缩在估算上下文用量超过自动压缩阈值时触发。普通自动压缩连续失败 3 次后进入熔断状态，停止普通自动压缩重试，并向用户提示需要手动处理。
- F10: 强制压缩在估算上下文用量超过强制压缩阈值时触发。强制压缩检查优先级高于普通自动压缩熔断状态；即使普通自动压缩已经熔断，越过强制压缩线时仍必须执行一次强制压缩。
- F11: 如果强制压缩经过重试后仍失败，LunaCode 停止当前请求并提示用户手动处理，不继续把接近硬上限的请求发送给 Provider。
- F12: 用户输入 `/compact` 时触发手动压缩。手动压缩是本地命令，不进入普通 Agent 对话，不自动让模型回复；压缩完成后在 TUI 显示压缩结果，包括覆盖消息数量、外置工具结果数量、恢复文件数量和失败原因。
- F13: 手动压缩与自动压缩使用同一套压缩逻辑。区别只在触发方式：自动压缩由系统检测阈值触发，手动压缩由用户主动命令触发。手动压缩即使当前历史尚未逼近窗口上限，也会尝试把较早历史压缩为摘要，用于从旧任务切换到新任务或提前腾出空间。
- F14: 重量压缩调用当前对话使用的 Provider 和模型配置，不在本阶段引入单独摘要模型。
- F15: 重量压缩目标是让压缩后的估算上下文用量低于自动压缩阈值。自动压缩、强制压缩和手动压缩都使用该目标预算。
- F16: 重量压缩把较早消息压缩为一条新的 user 摘要消息，并把近期消息原样保留在摘要消息之后。摘要消息包含早期摘要、完整会话记录路径、压缩边界提示和恢复文件快照，各部分用清晰分隔符分隔；恢复文件快照作为摘要消息的一部分拼接，不单独生成消息。
- F17: 近期原文保留规则为从尾部向前保留约 10,000 token 或至少 5 条消息，取覆盖范围更大的结果。保留边界不得从中间切断工具调用和工具结果的配对关系；必要时向前扩展边界。
- F18: 近期原文消息必须一字不改保留在对话中。模型最近做了什么、刚刚说了什么、刚刚收到哪些工具结果，都以原话形式保留，而不是摘要改写。
- F19: 重量压缩后的摘要消息必须包含固定结构，依次覆盖：主要请求和意图、关键技术概念、文件和代码段、错误和修复、问题解决过程、所有用户消息、待办任务、当前工作、可能的下一步。
- F20: 摘要中“所有用户消息”部分必须保留用户说过的所有非工具结果消息原文，不改写、不摘要、不遮蔽。
- F21: 摘要 Prompt 必须明确禁止模型调用任何工具。摘要 Prompt 必须要求模型先写分析草稿再写正式摘要；草稿只用于生成摘要，不写入压缩后的对话历史。
- F22: 压缩后必须插入边界提示，说明本次会话延续自之前对话，早期历史已摘要，近期消息已原样保留；如果需要具体代码片段、报错或工具细节，必须读取完整会话记录或重新读取项目文件，不得根据摘要脑补。
- F23: 如果一次会话发生多次压缩，旧摘要作为早期上下文的一部分参与新摘要。压缩后对话历史中仍只保留一条最新摘要消息，避免摘要消息持续堆叠。
- F24: 最近访问文件按真实文件工具成功访问记录判定，只统计成功访问过的文件，按最近访问时间取最多 5 个文件。恢复时读取文件当前内容，每个文件最多恢复约 5,000 token。
- F25: 如果压缩前使用过 Skill，压缩后应重新注入已使用 Skill 的定义，总预算为 25,000 token。Skill 定义恢复用于让模型在摘要后的首次请求中仍知道已启用 Skill 的核心规则。
- F26: 压缩后的首次模型请求仍应携带完整可用工具声明，确保模型不会因为历史压缩丢失工具使用能力。
- F27: 压缩模块不负责处理连续 user 消息的交替规则。摘要消息之后可以紧接保留的近期 user 原文消息；发送给 Provider 前的序列化层负责按 Provider 要求合并连续 user 消息。
- F28: 摘要请求自身过大并返回 Prompt Too Long 时，LunaCode 按 API 轮次分组待摘要消息，丢弃最旧的若干组后重试；最多重试 3 次。如果仍失败，继续丢弃 20% 的消息组再尝试，直到成功或达到强制失败处理条件。
- F29: Token 估算本阶段使用近似策略。LunaCode 锚定上一次 Provider 返回的 API usage，对新增消息按字符数近似估算，不要求引入精确 tokenizer。
- F30: 完整会话记录和外置工具结果可以原文落盘，以保留调试细节。TUI 提示、工具结果预览和摘要之外的用户可见短提示应遮蔽 API Key、Token、Authorization 等已知敏感值。
- F31: 摘要中的用户原始消息按原文保留，不因敏感值遮蔽策略被改写。LunaCode 应在配置或说明中提示用户，会话临时记录和摘要可能包含用户输入过的敏感内容。
- F32: 本阶段不因上下文压缩改变已有 Agent Loop 的正常行为。未触发压缩时，普通对话、工具调用、权限判断、MCP 工具调用、事件展示和中文协作风格应保持不变。

## 非功能需求

- N1: 压缩决策必须稳定、可解释。相同配置、相同消息历史和相同 Provider usage 下，应得到相同的压缩触发判断。
- N2: 轻量预防必须优先保护用户原始消息。用户消息尽量原文保留，不因为工具结果过大而被摘要改写。
- N3: 外置工具结果、会话记录和摘要边界提示必须足够清晰，让模型知道如何重新读取细节，而不是凭摘要补全代码。
- N4: 压缩失败处理必须避免死循环。普通自动摘要连续失败 3 次后熔断；强制压缩失败后停止当前请求并提示用户。
- N5: 压缩过程不能无限阻塞 Agent Loop 或 TUI。摘要调用、文件读取、文件写入和重试流程都应有可观察的进度或失败结果。
- N6: 压缩后的历史必须保持 Provider 可接受的消息结构。工具调用和工具结果配对不能被切断，连续同角色消息可以由序列化层处理。
- N7: 临时文件路径、会话记录路径和外置结果路径应位于项目内会话临时目录，并避免被误提交到版本控制。
- N8: 最近访问文件恢复应基于真实成功访问记录，而不是自然语言中出现的路径猜测，避免注入不相关文件。
- N9: Token 近似估算允许有误差，但必须保留安全余量，并在 Provider usage 返回后持续校准估算状态。
- N10: 本阶段不改变 LunaCode 默认中文协作风格。用户可见的压缩提示、失败提示和手动处理建议应使用中文。

## 不做的事

- 本阶段不实现跨会话记忆系统。
- 本阶段不实现项目级长期记忆、用户级全局记忆或可检索知识库。
- 本阶段不实现精确 tokenizer。
- 本阶段不实现摘要策略的机器学习优化或自适应学习。
- 本阶段不引入独立摘要模型配置。
- 本阶段不承诺重启后恢复外置工具结果、会话临时记录或压缩状态。
- 本阶段不把恢复文件快照作为长期缓存；恢复文件内容以压缩时重新读取到的当前内容为准。
- 本阶段不通过自然语言路径猜测恢复最近访问文件。
- 本阶段不改变工具权限策略、MCP 协议能力或 Provider 工具声明协议。
- 本阶段不自动清理用户历史项目数据之外的文件；会话临时目录清理策略只覆盖本阶段生成的当前会话临时产物。

## 验收标准

- AC1: 配置中声明上下文窗口后，LunaCode 能按“上下文窗口 - 摘要输出预留 - 自动压缩安全余量”计算自动压缩阈值，并按“自动压缩阈值 + 10,000”计算强制压缩阈值。
- AC2: 使用默认配置时，单个工具结果超过 50,000 字符会被写入项目内会话临时目录；对话里的工具结果只保留预览、完整路径和重新读取提示。
- AC3: 单条工具结果消息合计超过 200,000 字符时，LunaCode 会从最大的工具结果开始依次外置，直到剩余内容降到聚合限制以内。
- AC4: 每次模型请求前都能观察到轻量预防先运行，随后才判断是否需要自动摘要、强制摘要或继续普通请求。
- AC5: 压缩前会生成当前会话完整记录文件；压缩摘要消息中包含该记录文件路径，并提示模型需要细节时读取记录或重新读取项目文件。
- AC6: 估算上下文用量超过自动压缩阈值时，LunaCode 自动触发重量摘要，并在成功后把较早历史替换为一条结构化 user 摘要消息。
- AC7: 普通自动摘要连续失败 3 次后，LunaCode 不再反复自动重试，并给出中文提示说明自动摘要已熔断。
- AC8: 估算上下文用量超过强制压缩阈值时，即使普通自动摘要已经熔断，LunaCode 仍执行强制压缩。
- AC9: 强制压缩经过重试仍失败时，LunaCode 停止当前请求，不继续调用 Provider，并向用户提示需要手动处理上下文。
- AC10: 用户输入 `/compact` 时，LunaCode 执行本地手动压缩，不把 `/compact` 当成普通用户请求发送给模型；完成后 TUI 显示压缩结果摘要。
- AC11: 手动压缩即使在未超过自动压缩阈值时也会尝试压缩较早历史，适用于用户从一个任务切换到另一个任务或准备读取大量文件前主动腾出空间。
- AC12: 压缩后保留的近期消息至少覆盖约 10,000 token 或至少 5 条消息中更大的范围。
- AC13: 压缩边界不会切断工具调用和工具结果配对；若边界落在配对中间，LunaCode 会向前扩展保留范围。
- AC14: 近期原文消息在压缩后一字不改保留；对比压缩前后的近期消息内容，应完全一致。
- AC15: 压缩摘要包含 9 个固定部分：主要请求和意图、关键技术概念、文件和代码段、错误和修复、问题解决过程、所有用户消息、待办任务、当前工作、可能的下一步。
- AC16: 摘要中“所有用户消息”部分保留用户非工具结果消息原文，不摘要、不改写、不遮蔽。
- AC17: 摘要 Prompt 明确禁止工具调用，并要求模型先写分析草稿再写正式摘要；压缩后的对话历史中不存在分析草稿内容。
- AC18: 压缩后的摘要消息包含边界提示，明确要求模型需要具体代码片段、报错或工具细节时读取完整会话记录或重新读取项目文件，不得根据摘要脑补。
- AC19: 多次压缩后，对话历史中仍只保留一条最新摘要消息；最新摘要融合了旧摘要和新一段被压缩历史。
- AC20: 最近访问文件恢复只来自成功文件工具访问记录，最多恢复 5 个文件，每个文件最多约 5,000 token。
- AC21: 压缩前使用过 Skill 时，压缩后的首次请求能重新获得已使用 Skill 定义，总预算不超过 25,000 token。
- AC22: 压缩后的首次请求仍携带完整可用工具声明，模型仍能正常选择和调用工具。
- AC23: 摘要请求出现 Prompt Too Long 时，LunaCode 会按 API 轮次分组丢弃最旧消息组并重试；最多重试 3 次后仍失败时，继续丢弃 20% 消息组再试，直到成功或进入失败处理。
- AC24: Token 估算能使用上一次 Provider API usage 校准，并对新增消息按字符数近似估算；不依赖精确 tokenizer。
- AC25: 完整会话记录和外置工具结果原文落盘；TUI 短提示和工具结果预览中不会完整展示 API Key、Token 或 Authorization 等已知敏感值。
- AC26: 压缩功能启用后，未触发压缩的普通对话、工具调用、权限确认、MCP 工具调用和中文回复行为保持可用。
- AC27: 使用 tmux 启动 LunaCode，先让 Agent 读取多个文件并产生大工具结果，再输入 `/compact`，可以观察到 TUI 显示压缩完成、摘要消息生成、外置结果文件路径可读、近期原文仍保留。
- AC28: 使用 tmux 启动 LunaCode，在接近自动压缩阈值时继续对话，可以观察到 LunaCode 自动压缩；连续模拟摘要失败 3 次后普通自动压缩熔断；继续增长到强制压缩线时仍触发强制压缩。
- AC29: 使用 tmux 启动 LunaCode，压缩后输入“继续刚才的任务”，可以观察到模型基于摘要和近期原文继续工作；当需要具体旧代码细节时，会读取会话记录或重新读取项目文件，而不是直接编造摘要里没有的代码。
