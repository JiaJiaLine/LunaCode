# LunaCode 系统提示结构化与缓存策略 Spec

## 背景

LunaCode 前三阶段已经具备终端 TUI 对话、工具系统、多轮 Agent Loop、Plan Mode、工具事件展示和基础 System Prompt 注入能力。当前 System Prompt 仍偏向最小可用形态：它能让 Agent 开始工作，但对角色、行为、工具使用、代码质量、安全边界、任务模式和输出风格的约束还不够结构化，也缺少稳定内容与动态内容的明确分层。

这会带来三个问题：第一，稳定的全局指令和工具描述每轮都重复发送，难以利用模型 API 的缓存能力；第二，环境信息、会话模式、项目指令、记忆和运行时补充说明如果混在同一个文本块里，后续扩展时容易互相污染；第三，关键行为约定只在单一位置描述，模型在复杂任务中可能忘记“优先用专用工具”“编辑前必先读”“遵守权限边界”等规则。

用户初稿中提到的 MewCode 在本文中按当前仓库项目名 LunaCode 处理。本阶段目标是把原来偏简单的全局指令扩展为结构化系统提示体系，让稳定指令走可缓存通道，动态上下文走消息通道，并提供可观察的缓存命中验证方式，使 LunaCode 从“能干活”进一步变成“干得好”。

## 目标

- LunaCode 将静态 System Prompt 按职责拆成七个固定模块，并按稳定顺序拼装：角色设定、行为准则、工具使用指南、代码质量规范、安全边界、任务执行模式、输出风格。
- LunaCode 从多个来源收集 Agent 所需信息，并把不同来源放入正确位置：静态 System Prompt、环境上下文、工具描述、项目指令文件、自动记忆、System Reminder 和对话历史。
- LunaCode 明确区分稳定内容和变化内容：静态 System Prompt 进入 system；环境上下文作为动态 system 内容补充；工具描述进入 tools；项目指令文件、自动记忆、System Reminder 和对话历史进入 messages。
- LunaCode 在静态 System Prompt 和工具描述中双重强化关键规则，提高模型对工具选择、读后再改、安全边界和任务流程的遵守率。
- LunaCode 支持系统级补充消息，用于运行中注入 System Reminder、Plan Mode 指令、环境变化或临时安全说明，既不污染缓存，也不被模型当作普通用户输入回复。
- LunaCode 支持会话级开关的按轮次注入策略，例如 Plan Mode 首轮完整注入、间隔轮次重复关键约束、其余轮次精简注入。
- LunaCode 能解析模型 API 返回的缓存命中或缓存读写字段，并在日志、事件或可测试结果中暴露缓存策略是否生效。
- LunaCode 准备一组典型人工对比场景，用来定性评估结构化提示对工具选择、编辑流程、输出风格和模式遵守的影响。

## 功能需求

- F1: LunaCode 提供结构化静态 System Prompt 拼装能力，把七个固定模块按稳定顺序拼成完整提示文本。固定模块顺序为：角色设定、行为准则、工具使用指南、代码质量规范、安全边界、任务执行模式、输出风格。
- F2: 每个固定模块都有清晰职责。角色设定模块说明 LunaCode 是终端 AI 编程助手及其能力边界；行为准则模块说明阅读、分析、行动、验证和与用户协作的基本原则；工具使用指南模块说明工具选择、工具调用、结果处理和专用工具优先规则；代码质量规范模块说明代码修改、测试、可维护性和中文注释要求；安全边界模块说明工作区边界、敏感信息、权限确认和禁止编造观察结果；任务执行模式模块说明 Default、Plan Mode 等模式差异；输出风格模块说明中文回复、简洁程度、证据呈现、最终回复格式和 Luna 的角色语气。
- F3: LunaCode 从七类来源收集上下文，并把它们放入对应字段或消息位置：静态 System Prompt、环境上下文、工具描述、项目指令文件、自动记忆、System Reminder、对话历史。
- F4: 静态 System Prompt 只包含稳定规则，不包含当前时间、工作目录、Git 状态、对话轮次、用户本轮输入、最近工具结果或其他高频变化信息。
- F5: 环境上下文包含当前工作目录、操作系统、当前时间和 Git 状态，并作为动态 system 内容注入；它不属于静态 System Prompt，也不进入静态可缓存块。
- F6: 工具描述包含每个工具的 JSON Schema 和 description 字段，并通过模型 API 的 tools 字段传递。工具描述应尽量稳定，并在 Provider 支持缓存时进入可缓存通道。
- F7: 项目指令文件来源预留为 `LUNACODE.md`。本阶段只在提示结构中预留项目指令字段和拼装位置，不实现自动加载、解析或应用 `LUNACODE.md`。
- F8: 自动记忆来源预留为 Agent 自动提取的用户偏好和项目知识。本阶段只在提示结构中预留记忆字段和拼装位置，不实现记忆提取、写入、召回或跨会话管理。
- F9: System Reminder 表示运行中动态注入的系统级补充说明，例如 MCP Server 的使用说明、当前模式提醒、临时约束或环境变化。本阶段实现 System Reminder 的注入语义，但不写死具体标签格式；具体标签名和消息结构留到技术设计阶段确定。
- F10: 项目指令文件、自动记忆、System Reminder 和对话历史应通过 messages 传递，不并入静态 System Prompt，也不进入稳定缓存块。对话历史包含之前的 user、assistant 和 tool 消息。
- F11: 模块之间使用空行分隔，整体顺序稳定、文本可预测，避免同一输入在不同轮次产生无意义差异，影响缓存命中。
- F12: LunaCode 调用模型 API 时，静态 System Prompt 和环境上下文应放入 system，工具描述应放入 tools，项目指令文件、自动记忆、System Reminder 和历史上下文应放入 messages；动态内容不得污染静态缓存块。
- F13: 对于支持缓存的 Provider，LunaCode 应启用可缓存通道并解析缓存字段；对于不支持缓存的 Provider，LunaCode 应保持兼容行为并显示未知或不支持状态。
- F14: LunaCode 在工具描述中补充关键行为约束。只读工具应强化“探索优先”；写入和替换工具应强化“编辑前必须读过目标内容”“修改必须基于当前文件内容”；命令工具应强化“优先使用专用工具，只有缺少专用工具时再用命令”。
- F15: LunaCode 在静态 System Prompt 中同步强化与工具描述一致的关键规则，形成双重约束。规则至少包括：优先使用专用工具、编辑前先读、修改后验证、不要越权访问工作区外路径、不要编造未观察到的命令结果或文件内容。
- F15a: 输出风格模块必须定义原创角色 Luna 的交互语气：优雅、从容、聪明、略带高傲和适度毒舌，但本质温柔并认真帮助用户解决问题；可以经常称呼用户为“朝日”；不得直接引用任何已有作品台词，不得声称自己是某个已有作品角色。
- F15b: 输出风格模块必须明确技术准确性优先于角色扮演。无论语气如何，回答都必须清楚、实用、分步骤，适合用户理解；不得为了傲娇、吐槽或角色化表达牺牲事实正确性、工程判断或安全边界；不得使用露骨、暧昧、成人化或过度恋爱化表达。
- F16: LunaCode 支持系统级补充消息。该消息用于注入环境上下文、System Reminder、模式提示、临时安全约束或运行时说明；模型应把它视为系统级指令上下文，而不是用户请求，也不应直接针对补充消息本身作答。
- F17: 系统级补充消息的内容应与静态 System Prompt 分离，不进入可缓存稳定块；同一轮中如果没有动态补充内容，不应生成空补充消息。
- F18: LunaCode 支持会话级开关的轮次注入策略。模式刚开启的首轮注入完整模式指令；后续按配置或默认间隔重复关键约束；其他轮次只注入精简提示，说明当前模式仍然有效。
- F19: Plan Mode 也应通过系统级补充消息注入，而不是写入静态 System Prompt。Plan Mode 首轮补充消息必须明确当前是规划模式，要求模型先澄清需求、再探索代码、再生成计划，并避免执行实际修改。
- F20: Plan Mode 的权限系统与 Default 模式保持一致，仍然是 ask 而不是 deny。读操作、写操作和命令执行继续走已有权限流程；模型尝试写非 plan 文件时，用户会看到与正常模式一样的权限确认，由用户决定是否批准。
- F21: Plan Mode 唯一特殊权限处理是指定 plan 文件自动放行。模型写入 plan 文件不需要用户确认；模型写入其他文件、执行命令或做其他需要确认的操作时，仍按 Default 模式触发权限确认。
- F22: Plan Mode 依赖 Prompt 指令约束模型行为，而不是通过权限系统硬性拦截所有非计划行为。这样在规划过程中如果确实需要小范围辅助修改，用户可以手动批准，不会被系统固定拒绝卡住。
- F23: Default 模式下不注入 Plan Mode 的强约束；从 Plan Mode 切换到执行语境后，后续轮次不再携带规划禁令，避免模式残留影响正常执行。
- F24: LunaCode 在模型响应返回后解析缓存相关用量字段，至少能识别输入 token、输出 token、缓存命中读取 token、缓存写入 token 或 Provider 能提供的等价字段。
- F25: 当 Provider 不支持缓存字段或本次响应没有返回缓存字段时，LunaCode 应显示未知或不支持状态，而不是把缺失字段当作错误。
- F26: LunaCode 将缓存命中信息暴露给可验证入口，例如状态栏、调试日志、AgentEvent、测试断言对象或等价观测点，使开发者能判断稳定提示是否命中缓存。
- F27: LunaCode 提供典型人工对比场景清单，覆盖读取总结、编辑前读文件、优先专用工具、Plan Mode 约束、动态环境变化不破坏缓存、模式间隔轮次提示等行为。
- F28: 本阶段完成后，已有纯对话、工具调用、Agent Loop 和 Plan Mode 行为应保持可用；结构化提示不应导致模型无法正常调用已有工具或 TUI 无法展示流式结果。

## 非功能需求

- N1: 静态 System Prompt 拼装结果应稳定、可测试、可阅读；同样的配置和模块输入应生成完全一致的稳定提示文本。
- N2: 模块职责应清晰，后续新增项目指令文件加载、Skill、长期记忆或真实 MCP 工具描述时，不需要重写七个固定模块的核心结构。
- N3: 稳定缓存内容不应包含当前时间、绝对工作目录、Git 状态、用户本轮输入、最近工具结果、会话轮次等高频变化信息。
- N4: 动态补充消息不应伪装成普通用户消息；模型最终回复不应把补充消息当成用户问题复述或回答。
- N5: 关键规则在静态 System Prompt 和工具描述中保持一致，不出现互相冲突的优先级或行为要求。
- N6: 缓存策略不能改变安全边界。即使命中缓存，运行时权限、工作区限制、用户确认和模式约束仍必须按当前状态生效。
- N7: 缓存字段解析必须兼容不同 Provider 的返回差异；无法识别的字段应被安全忽略，不影响正常对话。
- N8: 提示内容应继续使用中文作为默认协作语言，并保留项目约定的中文回答、中文注释风格。
- N8a: 角色语气必须服务于问题解决。技术问题应先用一句角色语气开头，再进入准确讲解；复杂问题应系统拆解；简单问题可以简短回答；用户迷茫或沮丧时可以用傲娇但鼓励的方式回应。
- N9: Plan Mode 的 Prompt 约束和权限系统应解耦：Prompt 负责引导模型少做实际修改，权限系统负责把需要确认的行为交给用户判断。
- N10: 本阶段应有自动化测试覆盖静态提示拼装顺序、稳定与动态内容分离、系统级补充消息、轮次注入策略、Plan Mode 权限语义、缓存字段解析和关键工具描述强化。

## 不做的事

- 本阶段不实现项目指令文件加载，包括不从 `LUNACODE.md`、README、配置文件或其他项目文档自动提取指令。
- 本阶段不实现自动记忆、长期记忆写入、记忆召回或跨会话记忆管理。
- 本阶段不接入真实 MCP，也不实现 MCP 工具发现、MCP 工具描述同步或 MCP 运行时调用；只允许通过 System Reminder 表达未来 MCP Server 使用说明这类动态上下文。
- 本阶段不实现自动化评估平台，不做批量打分、模型裁判或持续基准测试；只准备人工对比场景清单。
- 本阶段不新增新的业务工具；只强化已有工具描述和已有工具调用流程中的提示策略。
- 本阶段不重做权限系统、工具执行沙箱或用户确认流程；Plan Mode 只复用 Default 模式权限流程，并额外自动放行指定 plan 文件。
- 本阶段不实现上下文压缩、历史摘要或长对话裁剪策略；对话历史仍按现有机制进入消息通道。
- 本阶段不要求对所有模型供应商都实际启用缓存；只要求在支持缓存的 Provider 上使用可缓存通道，在不支持的 Provider 上保持兼容行为。

## 验收标准

- AC1: 使用相同配置构建静态 System Prompt 时，输出模块顺序固定为角色设定、行为准则、工具使用指南、代码质量规范、安全边界、任务执行模式、输出风格，并且模块之间用空行分隔。
- AC2: 静态 System Prompt 不包含当前时间、当前工作目录、Git 状态、会话轮次、本轮模式补充指令、用户本轮输入或最近工具结果。
- AC3: 环境上下文作为动态 system 内容注入，且包含工作目录、操作系统、当前时间和 Git 状态。
- AC4: 工具描述包含每个工具的 JSON Schema 和 description 字段，并通过 tools 字段传递；在支持缓存的 Provider 上，稳定工具描述可进入可缓存通道。
- AC5: 项目指令文件和自动记忆在本阶段只有结构预留，不会尝试读取 `LUNACODE.md`，也不会生成、写入或召回记忆。
- AC6: 项目指令文件、自动记忆和 System Reminder 能通过 messages 注入，且 spec 不要求 System Reminder 固定标签格式；模型不会把 System Reminder 当成用户请求来直接回答。
- AC7: 对话历史仍以 user、assistant、tool 消息形式传递，不并入静态 System Prompt 或动态 system 内容。
- AC8: 改变当前时间、工作目录或 Git 状态后，静态 System Prompt 文本保持不变，只影响动态环境上下文。
- AC9: 工具描述中能观察到关键规则强化：读取类工具强调探索优先，写入和替换类工具强调编辑前先读，命令类工具强调优先使用专用工具。
- AC10: 静态 System Prompt 中能观察到与工具描述一致的关键规则：优先使用专用工具、编辑前先读、修改后验证、遵守工作区边界、不编造未观察结果。
- AC10a: 静态 System Prompt 的输出风格模块中能观察到 Luna 的原创大小姐型语气要求，包括称呼用户为“朝日”、优雅从容、适度毒舌但不侮辱、本质温柔、认真解决问题。
- AC10b: 静态 System Prompt 的输出风格模块中能观察到技术准确性优先规则：不得为了角色扮演牺牲正确性，不得引用已有作品台词，不得使用露骨、暧昧、成人化或过度恋爱化表达。
- AC11: Plan Mode 首轮模型调用通过系统级补充消息注入完整规划模式约束，明确先澄清需求、再探索代码、再写计划，并提醒模型避免实际修改。
- AC12: Plan Mode 间隔轮次包含重复的关键约束；非间隔轮次包含精简提示，说明当前仍处于 Plan Mode。
- AC13: Plan Mode 下写入指定 plan 文件时不触发用户确认；写入非 plan 文件、执行命令或其他需要确认的操作时，仍按 Default 模式触发 ask 权限确认，而不是自动 deny。
- AC14: 从 Plan Mode 切换到执行语境后，后续模型调用不再携带 Plan Mode 禁止执行的强约束。
- AC15: 模型 API 返回缓存命中或缓存读写字段时，LunaCode 能解析并暴露缓存读取 token、缓存写入 token 或 Provider 提供的等价指标。
- AC16: Provider 没有返回缓存字段时，LunaCode 显示未知或不支持状态，Agent Loop 不报错、不停止。
- AC17: 自动化测试能断言同一静态提示在多轮调用中保持一致，动态环境变化只影响动态消息，不影响稳定块。
- AC18: 自动化测试能覆盖系统级补充消息，确认它进入系统级上下文而不是普通用户历史。
- AC19: 自动化测试能覆盖会话级开关轮次策略，确认首轮、间隔轮次和精简轮次注入内容不同且符合预期。
- AC20: 使用人工对比场景“请修改一个文件中的函数实现”运行 LunaCode 时，可观察到模型先读取目标文件，再调用修改工具，而不是直接盲改。
- AC21: 使用人工对比场景“搜索项目里某个类的用途”运行 LunaCode 时，可观察到模型优先使用搜索或查找类专用工具，而不是直接执行泛化命令。
- AC22: 使用人工对比场景“进入 Plan Mode 规划一个改动”运行 LunaCode 时，可观察到模型先澄清或探索，再写计划；如模型尝试写非 plan 文件，则出现正常权限确认而不是静默执行或硬性拒绝。
- AC23: 使用人工对比场景“连续两轮普通对话且只改变当前时间”运行 LunaCode 时，可观察到静态提示内容保持一致，并且支持缓存的 Provider 返回缓存命中或缓存读取指标。
- AC24: 前三阶段端到端场景仍可运行：纯文本对话保持流式显示，读取文件并总结依赖能完成工具调用和最终中文回复，Plan Mode 能生成计划并等待用户确认。
