# LunaCode 项目记忆与会话恢复 Spec

## 背景

LunaCode 已经具备终端 TUI 对话、工具调用、权限系统、MCP 工具接入、结构化 Prompt 和上下文压缩能力。当前仍缺少稳定的项目知识、用户偏好和会话历史恢复机制：新会话启动后，Agent 主要依赖当前上下文，容易忘记项目约定、用户偏好和上次中断位置。

本阶段要让 LunaCode 在启动和每轮请求前自动恢复必要上下文：项目指令文件提供显式规则，会话 JSONL 提供可恢复的工作记忆，自动记忆提供长期偏好和项目知识。目标是让 Agent 从「每次失忆」变成「越用越懂你」，同时保留明确边界，避免引入向量数据库、RAG 检索或团队记忆同步。

## 目标

- LunaCode 启动时自动加载项目指令、用户偏好、项目记忆索引和最近未过期会话。
- LunaCode 会话以项目级 JSONL 文件追加保存，崩溃时最多丢失最后一行，恢复时能跳过坏行并处理未完成工具调用。
- LunaCode 自动恢复当前项目最近一个未过期会话，同时支持用户通过 `/session` 命令查看、切换和新建会话。
- LunaCode 在每轮 Agent Loop 自然结束后，异步更新长期记忆，不阻塞用户下一轮输入。
- LunaCode 支持用户级记忆和项目级记忆分层保存，并在处理请求前把记忆索引注入上下文。
- LunaCode 通过 `/memory` 命令提供记忆查看、列出、删除和自动记忆开关能力。
- LunaCode 在 TUI 状态栏中展示当前会话短标识、自动记忆开关和最近一次记忆更新状态。

## 功能需求

- F1: LunaCode 支持三层项目指令文件加载。项目根目录 `LUNACODE.md` 优先级最高，项目根目录 `.lunacode/LUNACODE.md` 次之，用户级 `~/.lunacode/LUNACODE.md` 最低；高优先级内容在注入上下文时排在低优先级内容之前。
- F2: 指令文件支持 `@include` 引用。系统逐行扫描内容，当某一行包含可识别的 `@include` 指令时，解析路径、读取目标文件，并用目标文件展开后的内容替换原行。
- F3: `@include` 支持递归处理。被引用文件中的 `@include` 继续展开；系统必须用最大深度 5 作为兜底限制，并维护已处理绝对路径集合，遇到重复路径直接跳过，防止环路。
- F4: `@include` 路径按来源根限制。项目级指令只能引用项目根目录内文件；用户级指令只能引用 `~/.lunacode/` 内文件；解析后的真实路径跳出允许根时必须跳过并记录原因。
- F5: 会话文件保存在项目根目录 `.lunacode/sessions/` 下，每个会话一个 JSONL 文件，文件名格式为 `YYYYMMDD-HHMMSS-xxxx.jsonl`，其中末尾 4 位为十六进制随机后缀，用于避免同秒冲突。
- F6: 每条会话 JSONL 记录只包含 `role`、`content`、`ts` 三个字段。`role` 对应 Claude API 风格消息角色；`content` 可以是字符串，也可以是 content block 数组；`ts` 是 Unix 时间戳。
- F7: 会话写入采用追加写。新增用户消息、assistant 最终消息、工具调用和工具结果都追加到当前会话 JSONL；崩溃或中断后，恢复时坏行可跳过，不应导致整个会话不可恢复。
- F8: LunaCode 不单独维护会话 meta 文件。需要会话 ID、标题、消息数、最后活跃时间和是否过期时，系统直接扫描 JSONL 文件计算；标题从第一条用户消息截断生成。
- F9: LunaCode 启动时自动清理 `.lunacode/sessions/` 下超过 30 天的会话文件，并通过可见事件或状态提示报告清理结果。
- F10: LunaCode 启动时自动恢复当前项目最近一个未过期会话。如果没有可恢复会话，则创建新会话。
- F11: LunaCode 支持 `/session` 基础会话闭环：查看当前会话、列出历史会话、恢复指定会话、新建会话。
- F12: 会话恢复时跳过无法解析的坏行；如果末尾存在未配对的 `tool_use` content block，系统从包含该 `tool_use` 的 assistant 消息开始截断，避免恢复到协议不完整状态。
- F13: 会话恢复时如果历史内容超过当前上下文预算，系统先触发现有压缩能力压缩一次；压缩失败时不修改原 JSONL，不阻塞启动，只在当前上下文中保留一条会话摘要提示。
- F14: 如果恢复会话距离上次活跃超过 24 小时，系统在对话中插入一条时间跨度提醒，说明上次会话时间、期间可能有代码变更，并建议 Agent 重新读取相关文件后再决策。
- F15: 自动记忆分为四类：用户偏好、纠正反馈、项目知识、参考信息。用户偏好和纠正反馈写入用户级目录 `~/.lunacode/memory/`；项目知识和参考信息写入项目级目录 `<项目根>/.lunacode/memory/`。
- F16: 每条记忆是带 frontmatter 的 Markdown 文件。frontmatter 至少包含 `id`、`type`、`title`、`created_at`、`updated_at`、`source_session`；正文保存记忆内容。`type` 决定记忆写入用户级目录还是项目级目录。
- F17: 用户级和项目级记忆各自维护一份索引文件，文件名均为 `MemoryIndex.md`。请求前注入上下文时，系统合并用户级和项目级索引，合并后的索引上下文控制在 200 行或 25KB 以内。
- F18: 自动记忆默认可由配置 `memory.auto_update` 控制。运行时 `/memory on` 和 `/memory off` 可以切换自动记忆状态，至少影响当前会话。
- F19: LunaCode 支持 `/memory` 基础管理闭环：查看记忆索引、列出记忆、删除指定记忆、开启自动记忆、关闭自动记忆。
- F20: 每轮 Agent Loop 自然结束后，也就是模型给出最终回复且不再调用工具时，系统异步触发自动记忆更新。异步任务不阻塞用户下一轮输入。
- F21: 自动记忆更新复用当前 Provider 配置调用模型。输入包含本轮对话增量、工具摘要、最终回复和现有相关记忆索引；系统不要求逐字逐句检测整个历史会话。
- F22: 自动记忆更新模型可以请求新增、更新、删除或保持不变。去重和是否需要改写由模型判断；系统负责校验输出、过滤敏感值、按 `type` 落盘并更新 `MemoryIndex.md`。
- F23: 自动记忆更新失败时不影响用户已收到的最终回复，不写入正常聊天回复，只通过轻量状态事件或状态栏提示失败原因摘要。
- F24: Prompt 注入顺序为项目指令、记忆索引、恢复会话历史。项目指令和记忆索引在处理请求前注入，使 Agent 相当于已经读过这些上下文。
- F25: TUI 状态栏展示当前 session ID 短标识、自动记忆是否开启、最近一次记忆更新状态。恢复失败、记忆失败和过期会话清理等情况也应有可观察状态。

## 非功能需求

- N1: 会话恢复和记忆注入必须保守失败。坏行、坏记忆文件、坏索引或 include 失败不得导致 LunaCode 启动失败或主对话不可用。
- N2: 会话 JSONL 追加写应具备崩溃友好性，恢复时最多丢弃或跳过损坏部分，不应要求额外 meta 文件保持同步。
- N3: 项目指令、记忆索引和恢复历史的总量必须受上下文预算约束；索引注入必须遵守 200 行 / 25KB 上限。
- N4: 敏感值不得被完整写入记忆、索引、状态事件或普通回复。自动记忆落盘前必须做敏感值过滤。
- N5: 用户级记忆不得复制到项目目录；项目级记忆不得写入用户级目录。记忆落点由 frontmatter 的 `type` 稳定决定。
- N6: 自动记忆更新必须异步执行，不能阻塞用户下一轮输入，也不能因为失败打断 Agent Loop。
- N7: 会话恢复、会话命令、记忆命令和状态栏展示应使用中文可理解提示，保持 LunaCode 现有中文协作风格。
- N8: 指令文件 include 展开必须跨平台处理相对路径、绝对路径、规范化路径和符号链接后的真实路径。
- N9: 会话恢复必须保持工具调用协议完整性。不能把未配对工具调用恢复给模型。
- N10: 本阶段能力应有自动化测试覆盖，包括指令加载、include 防环、JSONL 恢复、坏行跳过、工具调用截断、过期清理、记忆落点、索引上限、命令行为和异步失败处理。

## 不做的事

- 本阶段不实现向量数据库。
- 本阶段不实现 RAG 检索或相似度召回。
- 本阶段不实现团队记忆同步、远程记忆共享或多用户协作记忆。
- 本阶段不实现用户自定义记忆分类；分类固定为用户偏好、纠正反馈、项目知识、参考信息四类。
- 本阶段不实现记忆全文搜索、编辑器式记忆编辑、记忆重建索引的高级交互。
- 本阶段不实现会话重命名、删除、导出或归档；会话命令只覆盖当前会话、列表、恢复和新建。
- 本阶段不实现 LLM 会话标题生成；会话标题从第一条用户消息截断得到。
- 本阶段不支持 glob include，也不允许 include 读取来源根之外的文件。
- 本阶段不把压缩失败的历史会话强行塞入上下文，也不因为恢复失败阻止 LunaCode 启动。

## 验收标准

- AC1: 同时存在三层 `LUNACODE.md` 时，LunaCode 请求前注入的项目指令按项目根、项目 `.lunacode`、用户级的优先级顺序排列。
- AC2: 指令文件中包含 `@include` 行时，LunaCode 能把该行替换为目标文件内容；目标文件中的 include 继续展开，最大深度为 5。
- AC3: 两个指令文件互相 include 时，LunaCode 不进入死循环，重复绝对路径被跳过，并能继续启动。
- AC4: 项目级 include 指向项目根外路径、用户级 include 指向 `~/.lunacode/` 外路径时，LunaCode 跳过该引用并记录中文原因。
- AC5: 新建会话时，`.lunacode/sessions/` 下生成 `YYYYMMDD-HHMMSS-xxxx.jsonl` 格式文件，后续消息以 JSONL 追加写入。
- AC6: JSONL 中每条记录都包含 `role`、`content`、`ts` 三个字段；assistant 的混合文本和工具调用可以通过 content block 数组保存。
- AC7: `/session` 能显示当前会话 ID、标题、消息数和最后活跃时间；标题来自第一条用户消息截断结果。
- AC8: `/session` 能列出历史会话、恢复指定会话和新建会话；恢复后后续消息写入被恢复的会话或新建会话。
- AC9: 启动时存在最近未过期会话时，LunaCode 自动恢复该会话；不存在可恢复会话时自动创建新会话。
- AC10: 启动时 `.lunacode/sessions/` 中存在超过 30 天的会话文件时，LunaCode 自动清理并显示可见状态提示。
- AC11: 会话 JSONL 中存在坏行时，恢复过程跳过坏行，其他合法消息仍能恢复。
- AC12: 会话末尾存在未配对 `tool_use` 时，恢复历史从该 assistant 消息开始截断，模型请求中不包含未配对工具调用。
- AC13: 恢复历史超过 token 预算时，LunaCode 先压缩一次；压缩失败时启动不失败，当前上下文只保留会话摘要提示。
- AC14: 恢复会话距离上次活跃超过 24 小时时，模型上下文中出现时间跨度提醒，提醒 Agent 重新读取相关文件。
- AC15: 用户偏好和纠正反馈类型记忆写入 `~/.lunacode/memory/`；项目知识和参考信息类型记忆写入 `<项目根>/.lunacode/memory/`。
- AC16: 新增记忆文件包含要求的 frontmatter 字段，`type` 与实际目录落点一致。
- AC17: 用户级和项目级 `MemoryIndex.md` 可被合并注入上下文；合并后的索引不超过 200 行或 25KB。
- AC18: `/memory` 能查看索引、列出记忆、删除指定记忆、开启自动记忆和关闭自动记忆。
- AC19: 配置 `memory.auto_update: false` 时，自动记忆不在 Agent Loop 结束后运行；通过 `/memory on` 打开后，后续自然结束的回合会触发异步记忆更新。
- AC20: 模型最终回复且没有继续工具调用时，LunaCode 在后台异步启动记忆更新；用户可以立即输入下一轮对话。
- AC21: 自动记忆更新可新增、更新、删除或保持不变，并在完成后更新对应 `MemoryIndex.md`。
- AC22: 自动记忆更新失败时，聊天回复不被追加失败文本；TUI 或事件流中能看到失败原因摘要。
- AC23: TUI 状态栏能显示当前 session 短标识、自动记忆开关状态和最近一次记忆更新状态。
- AC24: 记忆写入和索引内容不会完整暴露 API Key、Token、Authorization 值或其他已知敏感值。
- AC25: 使用 tmux 启动 LunaCode，输入一段真实请求并让 Agent 完成最终回复后，可以观察到会话 JSONL 被追加、状态栏显示当前 session、自动记忆异步更新；重启后最近未过期会话被自动恢复。
