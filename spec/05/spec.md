# LunaCode 权限系统 Spec

## 背景

LunaCode 前四阶段已经具备终端 TUI 对话、工具系统、多轮 Agent Loop、Plan Mode、人在回路确认点和结构化 System Prompt。当前权限能力仍偏基础：读工具通常直接放行，写文件、改文件和命令执行主要依赖简单询问，路径工具已有工作区边界校验，但尚未形成一套统一、分层、可配置、可解释的权限体系。

这会带来三个问题：第一，已知高危命令缺少不可配置绕过的硬拦截层，过度依赖模型自觉或用户确认；第二，文件路径、命令参数和工具级策略散落在工具实现与权限网关中，缺少统一决策结果；第三，用户无法用用户级、项目级和本地级规则表达自己的信任偏好，也无法在一次拒绝后让 Agent Loop 基于拒绝结果继续调整策略。

用户初稿中提到的 MewCode 在本文中按当前仓库项目名 LunaCode 处理。本阶段目标是给 LunaCode 增加一套五层防御的权限系统：危险操作黑名单、路径沙箱、可配置规则、多档权限模式和人在回路确认。权限系统要成为工具执行前的统一决策入口，并在拒绝工具调用时把结构化结果回灌给模型，而不是直接终止 Agent Loop。

## 目标

- LunaCode 在工具执行前先经过统一权限决策，所有工具调用都得到 allow、deny 或 ask 的明确结果。
- LunaCode 使用不可被配置放开的危险操作黑名单，在命令执行前硬拦截已知高危命令。
- LunaCode 对文件读写路径执行工作区沙箱校验，解析符号链接后再判断是否仍位于项目目录内，防止路径逃逸。
- LunaCode 支持用户级、项目级和本地级三层规则文件，规则可以按工具名、参数模式或路径模式声明 allow 或 deny。
- LunaCode 支持多档权限模式，让用户可以整体切换 default、acceptEdits、plan 和 bypassPermissions 等信任等级；具体规则优先于权限模式，权限模式只在没有规则命中时提供默认行为。
- LunaCode 在规则没有明确命中时触发人在回路确认，用户可以选择本次允许、始终允许或拒绝。
- LunaCode 在权限被拒绝时继续 Agent Loop，把拒绝原因作为结构化工具结果提供给模型，让模型有机会调整方案。
- LunaCode 保持已有 Plan Mode 的安全语义：权限矩阵与 default 完全一致，主要依赖 System Prompt 约束模型优先做只读规划。

## 功能需求

- F1: LunaCode 提供统一权限决策入口。工具执行前必须把工具名、工具参数、运行模式、工作区根目录、目标路径摘要和当前权限模式交给权限系统，权限系统返回 allow、deny 或 ask。
- F2: LunaCode 对所有工具执行统一应用权限决策。任何工具不能绕过权限系统直接执行；工具不存在或参数无法解析时仍返回结构化失败结果。
- F3: LunaCode 提供危险操作黑名单。黑名单以正则模式描述已知高危命令或命令片段，至少覆盖递归删除根目录或工作区外路径、强制清理系统目录、格式化磁盘、修改启动项、关闭关键安全机制、无限 fork 或等价资源炸弹等明显危险操作。
- F4: 危险操作黑名单是最高优先级的硬拒绝层。即使处于 bypassPermissions 模式、规则配置为 allow 或用户此前选择始终允许，命中黑名单的命令也必须被拒绝。
- F5: 危险操作黑名单只拦截可观察的高危命令，不负责网络访问限制、资源配额或审计日志。
- F6: LunaCode 对所有带路径参数的工具执行路径沙箱校验。相对路径必须基于当前工作区解析，绝对路径必须在解析后仍位于工作区内。
- F7: 路径沙箱必须先解析符号链接或最近存在父目录的真实路径，再做工作区前缀判断，防止通过符号链接、`..`、大小写差异或等价路径表示逃逸。
- F8: 路径沙箱区分读路径、写路径和命令工作目录。读写文件、搜索路径、glob 范围和命令 cwd 都必须被纳入校验；没有路径参数的工具不因路径沙箱失败。
- F9: LunaCode 支持可配置规则。规则写成“工具名(模式)”的形式，例如 `Bash(git *)` 表示匹配 Bash 工具中以 `git ` 开头的命令；每条规则结果只有 allow 或 deny。
- F10: 规则匹配支持精确匹配和 glob 匹配。工具名必须精确匹配；括号内模式可以匹配命令文本、路径文本或权限系统从参数中提取出的规范化摘要。
- F11: LunaCode 支持三层 YAML 规则文件。本地级规则文件位于项目根目录 `.lunacode/permissions.local.yaml`，用于个人覆盖，不提交到版本控制；项目级规则文件位于项目根目录 `.lunacode/permissions.yaml`，由项目维护者设置，可以提交到版本控制；用户级规则文件位于 `~/.lunacode/permissions.yaml`，作为所有项目通用的全局偏好。
- F12: 规则优先级按本地级、项目级、用户级逐层判断。本地级优先级最高，项目级次之，用户级最低。
- F13: 同一层中更具体的匹配优先于更宽泛的匹配；仍无法区分时按文件中靠后的规则覆盖靠前规则。
- F14: deny 规则在同一层中优先于 allow 规则，除非存在更具体的同层 allow 规则或更高层级 allow 规则明确覆盖。
- F15: LunaCode 支持四档权限模式。default 模式自动放行只读操作 ReadFile、Glob 和 Grep，写操作和命令执行需要用户确认；acceptEdits 模式在 default 的基础上自动放行文件编辑操作 WriteFile 和 EditFile；plan 模式的权限矩阵与 default 完全一致，写操作和命令执行仍需要用户确认；bypassPermissions 模式自动放行大多数操作，但危险操作黑名单仍然生效。
- F16: 具体规则优先于权限模式。权限系统先判断黑名单和路径沙箱等硬安全层，再按本地级、项目级、用户级查找具体规则；只有没有具体规则命中时，才使用当前权限模式给出默认 allow、ask 或 deny。
- F17: LunaCode 在规则没有明确命中，且权限模式也不能直接给出最终 allow 或 deny 时，触发人在回路确认。
- F18: 人在回路确认需要展示工具名、参数摘要、目标路径或命令摘要、命中的权限模式、未命中规则的原因，以及可选动作。
- F19: 人在回路确认支持本次允许、始终允许和拒绝。本次允许只放行当前工具调用；始终允许写入本地级 `.lunacode/permissions.local.yaml` 作为个人覆盖规则；拒绝返回结构化权限错误。
- F20: 始终允许只自动生成 allow 规则，不自动生成 deny 规则；用户可以手动编辑三层规则文件添加个人、项目或全局偏好。
- F21: 人在回路只负责权限确认，不替代需求澄清、Plan Mode 执行确认或 AskUserQuestion。
- F22: 权限被拒绝时，LunaCode 不终止 Agent Loop。系统把拒绝结果作为工具结果回灌给模型，包含错误类型、拒绝层级、可重试建议和被拒绝的安全原因摘要。
- F23: 如果模型连续多次请求被同一权限规则拒绝，Agent Loop 可以触发现有异常或迭代上限保护；权限系统本身不直接杀死进程或清空对话。
- F24: Plan Mode 的权限矩阵与 default 完全一致，写操作和命令执行都需要用户确认。Plan Mode 的约束主要由 System Prompt 引导模型自觉只做只读操作；如果模型尝试调用写工具或执行命令，权限系统仍按 default 级别触发 ask。
- F25: 只读工具默认仍可在路径沙箱通过后被较宽松地放行；写文件、改文件、命令执行和其他有副作用工具默认更倾向 ask 或 deny，具体行为由权限模式和规则共同决定。
- F26: LunaCode 能加载规则配置并在配置格式错误时给出可理解错误。错误规则不得导致权限系统整体失效为全放行；无法加载的规则层应被忽略或标记为无效，并进入保守行为。
- F27: LunaCode 提供可观察的权限决策结果。TUI 或 Agent 事件流能看到权限请求、权限允许、权限拒绝、命中规则、命中黑名单、沙箱拒绝或用户确认结果等状态。
- F28: 规则配置中的敏感路径、命令参数或环境变量值不应被完整泄露到普通回复中；展示给用户和模型的摘要应避免暴露 API Key、Token 或已知敏感值。
- F29: 本阶段完成后，已有纯对话、工具调用、Agent Loop、Plan Mode、结构化 Prompt 和工具事件展示行为应保持可用。

## 非功能需求

- N1: 权限决策顺序必须稳定、可测试、可解释。相同工具调用、相同规则和相同权限模式应得到一致结果。
- N2: 权限系统默认失败策略应保守。黑名单解析失败、路径真实路径解析失败、规则文件格式错误或权限确认异常时，不得静默全放行。
- N3: 路径沙箱必须跨平台工作，至少兼容 Windows 路径分隔符、盘符大小写、相对路径、绝对路径和符号链接。
- N4: 权限规则匹配应足够简单，避免引入脚本执行、表达式求值或可执行配置，防止规则文件本身成为攻击面。
- N5: 权限系统应与具体工具实现解耦。新增工具时，只要提供工具名、参数摘要和风险属性，就能进入同一套权限流程。
- N6: 权限系统应与 Provider 和 TUI 解耦。模型供应商差异不影响权限判断；TUI 只消费权限事件和确认请求。
- N7: 人在回路确认应避免阻塞整个应用到不可恢复状态。用户拒绝、取消或输入无法识别时，应得到明确结果并继续正常交互。
- N8: 权限被拒绝后的工具结果应足够简洁，既能让模型调整策略，又不泄露敏感配置或过多本地环境细节。
- N9: 规则加载和权限决策应有自动化测试覆盖，包括黑名单、沙箱、规则优先级、权限模式、人在回路和拒绝后继续循环。
- N10: 本阶段不改变项目默认中文协作风格；用户可见权限说明、错误摘要和规则示例应使用中文。

## 不做的事

- 本阶段不实现网络请求限制，不按域名、IP、端口或协议控制外联访问。
- 本阶段不实现 CPU、内存、磁盘、子进程数量、命令运行时长之外的新资源配额系统。
- 本阶段不实现审计日志、合规报表、不可篡改日志或长期安全事件查询。
- 本阶段不实现插件市场、第三方权限策略插件或动态下载规则。
- 本阶段不实现复杂策略语言、条件表达式、脚本化规则或基于时间、用户身份、Git 分支的高级策略。
- 本阶段不实现图形化规则管理界面；规则以 YAML 文件和人在回路确认入口维护。
- 本阶段不要求支持多用户协作权限、远程工作区权限同步或企业集中策略下发。
- 本阶段不改变模型工具调用协议，也不新增业务工具；只把已有工具执行纳入新的权限决策流程。
- 本阶段不要求权限系统识别所有可能危险命令。黑名单只覆盖明确高危模式，其他未知风险通过规则、权限模式和人在回路处理。

## 验收标准

- AC1: 任意工具执行前都会经过统一权限决策，并能观察到 allow、deny 或 ask 结果。
- AC2: Bash 命令命中危险操作黑名单时，即使存在 allow 规则或处于 bypassPermissions 模式，工具调用仍被拒绝。
- AC3: 黑名单拒绝结果包含明确错误类型和简短原因，并作为工具结果回灌给模型，而不是导致 Agent Loop 崩溃。
- AC4: 文件工具传入工作区内普通相对路径时，路径沙箱允许继续后续权限判断。
- AC5: 文件工具传入 `..` 逃逸路径、工作区外绝对路径或指向工作区外的符号链接时，路径沙箱拒绝该工具调用。
- AC6: 命令工具传入工作区外 cwd 时，路径沙箱拒绝该命令执行。
- AC7: 用户级规则 `Bash(git *) = allow` 能允许匹配的 git 命令进入后续权限模式判断。
- AC8: 项目级或本地级 deny 规则能覆盖用户级 allow 规则。
- AC9: 本地级 allow 规则能覆盖项目级或用户级默认规则，并且不会要求提交到版本控制。
- AC10: 同一层规则中，精确匹配优先于 glob 匹配；更靠后的同等规则覆盖更靠前规则。
- AC11: default 模式下，未被具体规则允许的写文件、改文件和命令执行会触发 ask，而不是直接执行。
- AC12: 默认模式下，只读工具在路径沙箱通过后可以顺畅执行；写文件、改文件和命令执行仍需要规则允许或用户确认。
- AC13: bypassPermissions 模式下，大多数未命中操作可以自动放行；危险操作黑名单仍然生效。
- AC14: 规则没有明确命中时，TUI 显示权限确认请求，包含工具名、参数摘要、目标路径或命令摘要和可选动作。
- AC15: 用户选择本次允许后，当前工具调用执行成功，但同类下一次调用仍会重新请求确认。
- AC16: 用户选择始终允许后，LunaCode 将 allow 规则写入 `.lunacode/permissions.local.yaml`，同一项目中匹配的后续工具调用不再重复询问。
- AC17: 重启 LunaCode 后，本地级 `.lunacode/permissions.local.yaml` 中由始终允许生成的规则仍能被加载。
- AC18: 用户拒绝权限请求后，工具不会执行，Agent Loop 继续运行，并把权限拒绝工具结果提供给模型。
- AC19: 模型在收到权限拒绝后可以改用更安全的工具或路径继续完成请求，而不是整轮请求直接中断。
- AC20: plan 模式下，写文件、改文件和命令执行的权限行为与 default 一致，会触发确认而不是自动放行。
- AC21: 规则文件格式错误时，LunaCode 给出可理解错误并进入保守行为，不会把该层规则当作全放行。
- AC22: 权限事件流或 TUI 中能观察到权限允许、权限拒绝、权限请求、黑名单命中、沙箱拒绝和用户确认结果。
- AC23: 权限提示、错误摘要和工具结果不会泄露配置中的 API Key、Token 或其他已知敏感值。
- AC24: 自动化测试覆盖黑名单硬拒绝、符号链接逃逸、规则优先级、规则优先于权限模式、人在回路本次允许、始终允许、拒绝后继续 Agent Loop。
- AC25: 使用 checklist 中指定的端到端场景运行 LunaCode，输入真实请求“读取当前项目的 pom.xml，然后尝试执行一个未配置允许的写入操作”，可观察到读取正常执行，写入触发权限确认，用户拒绝后 Agent Loop 继续给出中文解释和替代方案。
