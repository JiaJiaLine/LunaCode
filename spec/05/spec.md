# LunaCode 权限系统 Spec

## 背景

LunaCode 前四阶段已经具备终端 TUI 对话、工具系统、多轮 Agent Loop、Plan Mode、人在回路确认点和结构化 System Prompt。当前权限能力仍偏基础：读工具通常直接放行，写文件、改文件和命令执行主要依赖简单询问，路径工具已有工作区边界校验，但尚未形成一套统一、分层、可配置、可解释的权限体系。

这会带来三个问题：第一，已知高危命令缺少不可配置绕过的硬拦截层，过度依赖模型自觉或用户确认；第二，文件路径、命令参数和工具级策略散落在工具实现与权限网关中，缺少统一决策结果；第三，用户无法用用户级、项目级和本地级规则表达自己的信任偏好，也无法在一次拒绝后让 Agent Loop 基于拒绝结果继续调整策略。

用户初稿中提到的 MewCode 在本文中按当前仓库项目名 LunaCode 处理。本阶段目标是给 LunaCode 增加一套五层防御的权限系统：危险操作黑名单、路径沙箱、可配置规则、多档权限模式和人在回路确认。权限系统要成为工具执行前的统一决策入口，并在拒绝工具调用时把结构化结果回灌给模型，而不是直接终止 Agent Loop。

## 目标

- LunaCode 在工具执行前先经过统一权限决策，所有工具调用都得到 allow、deny 或 ask 的明确结果。
- LunaCode 使用不可被配置放开的危险操作黑名单，在命令执行前硬拦截已知高危命令。
- LunaCode 对文件读写路径执行路径沙箱校验，解析符号链接后再判断是否仍位于项目目录或显式配置的额外根目录内，防止路径逃逸。
- LunaCode 在 Linux 上对 Bash 命令提供 OS 级沙箱，通过 bubblewrap、seccomp 和 Linux namespace 创建隔离环境，并默认断网。
- LunaCode 支持用户级、项目级和本地级三层规则文件，规则可以按工具名、参数模式或路径模式声明 allow 或 deny。
- LunaCode 支持多档权限模式，让用户可以整体切换 default、acceptEdits、plan 和 bypassPermissions 等信任等级；具体规则优先于权限模式，权限模式只在没有规则命中时提供默认行为。
- LunaCode 在规则没有明确命中时触发人在回路确认，用户可以选择本次允许、始终允许或拒绝。
- LunaCode 在权限被拒绝时继续 Agent Loop，把拒绝原因作为结构化工具结果提供给模型，让模型有机会调整方案。
- LunaCode 保持已有 Plan Mode 的安全语义：权限矩阵与 default 完全一致，主要依赖 System Prompt 约束模型优先做只读规划。

## 功能需求

- F1: LunaCode 提供统一权限决策入口。工具执行前必须把工具名、工具参数、运行模式、工作区根目录、目标路径摘要和当前权限模式交给权限系统，权限系统返回 allow、deny 或 ask。
- F2: LunaCode 对所有工具执行统一应用权限决策。任何工具不能绕过权限系统直接执行；工具不存在或参数无法解析时仍返回结构化失败结果。
- F3: LunaCode 提供 Bash 危险操作黑名单。黑名单以正则模式描述已知高危 Bash 命令或命令片段，至少覆盖递归删除根目录或允许根目录外路径、强制清理系统目录、格式化磁盘、修改启动项、关闭关键安全机制、无限 fork 或等价资源炸弹等明显危险操作。文件工具不进入危险操作黑名单，文件访问风险由路径沙箱、敏感路径 ask、规则和权限模式处理。
- F4: 危险操作黑名单是最高优先级的硬拒绝层。即使处于 bypassPermissions 模式、规则配置为 allow 或用户此前选择始终允许，命中黑名单的命令也必须被拒绝。
- F5: 危险操作黑名单只拦截可观察的高危命令，不负责网络访问限制、资源配额或审计日志。
- F5a: LunaCode 在 Linux 上执行 Bash 命令时必须支持 OS 级沙箱。OS 级沙箱使用 bubblewrap 创建隔离环境，并结合 seccomp 限制危险系统调用；非 Linux 平台本阶段不要求实现等价 OS 级沙箱，但仍必须执行黑名单、路径沙箱、规则和人在回路权限判断。
- F5b: Linux OS 级沙箱默认断网。LunaCode 通过 bubblewrap 的 `--unshare-net` 创建空网络命名空间，使 Bash 命令默认不能访问网络。
- F5c: 用户确实需要命令访问网络时，可以通过配置 `sandbox.network_enabled: true` 开启网络。未显式开启时，网络必须保持关闭；开启网络只影响 OS 级沙箱网络命名空间，不绕过 Bash 黑名单、路径沙箱、deny 规则或人在回路确认。
- F5d: Bash 命令进入 OS 级沙箱后，只应挂载项目根目录、显式配置的额外根目录以及运行命令所需的最小系统路径；允许根目录之外的宿主文件系统不应暴露给命令。
- F6: LunaCode 对所有带路径参数的工具执行路径沙箱校验。允许根目录集合包含当前工作区和显式配置的额外根目录；相对路径默认基于当前工作区解析，绝对路径必须在解析后仍位于某个允许根目录内。
- F6a: 额外根目录通过 `sandbox.extra_roots` 配置声明，每个额外根目录必须包含稳定别名和本机真实路径；规则匹配和权限提示中使用 `/roots/{name}/...` 虚拟路径表示该额外根目录。
- F7: 路径沙箱必须按固定流程验证路径：先将请求路径转换为绝对路径并规范化；如果目标路径已经存在，解析符号链接得到真实路径；如果目标路径不存在，必须找到最近存在的父目录，解析该父目录的符号链接得到真实父目录，再拼接剩余路径片段得到待创建目标路径；最后检查真实路径或待创建目标路径是否仍位于允许根目录内。不存在的文件不能因为无法解析自身符号链接而直接放行。
- F8: 路径沙箱区分读路径、写路径和 Bash 路径。读写文件、搜索路径、glob 范围、Bash cwd 以及 Bash 命令文本中可识别的明显路径片段都必须被纳入校验；没有路径参数且命令文本中没有可识别路径片段的工具不因路径沙箱失败。本阶段不要求完整 shell 语义解析。
- F9: LunaCode 支持可配置规则。规则文件使用 YAML 顶层列表，每条规则包含 `rule` 和 `effect` 两个字段；`rule` 写成“工具名(模式)”的形式，例如 `Bash(git *)` 表示匹配 Bash 工具中以 `git ` 开头的命令；`effect` 只有 `allow` 或 `deny` 两种。规则文件允许使用 YAML 注释说明规则用途。
- F10: 规则匹配支持精确匹配和 glob 匹配。工具名必须精确匹配；括号内模式按工具匹配固定对象：`Bash(...)` 匹配命令文本，`ReadFile(...)` 匹配读取路径，`WriteFile(...)` 匹配写入路径，`EditFile(...)` 匹配编辑路径，`Glob(...)` 匹配 glob 入参或搜索范围，`Grep(...)` 匹配搜索 pattern；路径类匹配对象在规则匹配前统一规范化为虚拟根路径，项目目录使用 `/project/...`，额外根目录按配置别名映射为 `/roots/{name}/...`，路径范围仍必须额外通过路径沙箱校验。
- F11: LunaCode 支持三层 YAML 规则文件。本地级规则文件位于项目根目录 `.lunacode/permissions.local.yaml`，用于个人覆盖，不提交到版本控制；项目级规则文件位于项目根目录 `.lunacode/permissions.yaml`，由项目维护者设置，可以提交到版本控制；用户级规则文件位于 `~/.lunacode/permissions.yaml`，作为所有项目通用的全局偏好。
- F12: deny 规则跨层合并且不可翻转。权限系统会收集本地级、项目级和用户级三层中的所有 deny 规则；只要任意一层 deny 命中，最终结果就是 deny，其他层级或同层的 allow 都不能覆盖。
- F13: allow 规则在没有 deny 命中的前提下按本地级、项目级、用户级逐层判断。本地级 allow 优先级最高，项目级次之，用户级最低。
- F14: 同一层级内，后定义的规则优先级更高。权限系统按文件顺序保留规则，多个同层 allow 同时命中时采用靠后的 allow；多个同层 deny 同时命中时采用靠后的 deny 作为拒绝原因。该排序不允许 allow 覆盖任何已命中的 deny。
- F15: LunaCode 支持四档权限模式。default 模式自动放行只读操作 ReadFile、Glob 和 Grep，写操作和命令执行需要用户确认；acceptEdits 模式在 default 的基础上自动放行文件编辑操作 WriteFile 和 EditFile；plan 模式的权限矩阵与 default 完全一致，写操作和命令执行仍需要用户确认；bypassPermissions 模式自动放行大多数操作，但危险操作黑名单和路径沙箱仍然生效。权限模式可以由配置文件提供默认值，也必须支持运行时切换，运行时切换只影响当前会话。
- F16: 具体规则优先于权限模式。权限系统先判断危险操作黑名单和路径沙箱等硬安全层，再收集三层 deny 规则并执行不可翻转的拒绝判断，然后按本地级、项目级、用户级查找 allow 规则；只有没有具体规则命中时，才使用当前权限模式给出默认 allow、ask 或 deny。
- F16a: LunaCode 提供运行时权限模式切换入口。用户输入 `/permissions` 时显示当前权限模式；用户输入 `/permissions default`、`/permissions acceptEdits`、`/permissions plan` 或 `/permissions bypassPermissions` 时切换当前会话权限模式；切换到 bypassPermissions 必须进行二次确认，用户确认后才生效，取消则保持原权限模式；切换后后续工具调用使用新的权限模式，当前模式应在 TUI 状态栏、权限确认提示或 Agent 事件中可观察。
- F17: LunaCode 在规则没有明确命中，且权限模式也不能直接给出最终 allow 或 deny 时，触发人在回路确认。
- F18: 人在回路确认需要展示工具名、参数摘要、目标路径或命令摘要、命中的权限模式、未命中规则的原因，以及可选动作。
- F19: 人在回路确认支持本次允许、始终允许和拒绝。本次允许只放行当前工具调用；始终允许写入本地级 `.lunacode/permissions.local.yaml` 作为个人覆盖规则；拒绝返回结构化权限错误。
- F20: 始终允许只自动生成 allow 规则，不自动生成 deny 规则；自动生成的 allow 规则必须精确到本次工具参数，不主动放宽为同类命令或目录通配；用户可以手动编辑三层规则文件添加个人、项目或全局偏好。
- F21: 人在回路只负责权限确认，不替代需求澄清、Plan Mode 执行确认或 AskUserQuestion。
- F22: 权限被拒绝时，LunaCode 不终止 Agent Loop。系统把拒绝结果作为工具结果回灌给模型，包含错误类型、拒绝层级、可重试建议和被拒绝的安全原因摘要。
- F23: 如果模型连续多次请求被同一权限规则拒绝，Agent Loop 可以触发现有异常或迭代上限保护；权限系统本身不直接杀死进程或清空对话。
- F24: Plan Mode 的权限矩阵与 default 完全一致，写操作和命令执行都需要用户确认。Plan Mode 的约束主要由 System Prompt 引导模型自觉只做只读操作；如果模型尝试调用写工具或执行命令，权限系统仍按 default 级别触发 ask。
- F25: 只读工具默认仍可在路径沙箱通过后被较宽松地放行；写文件、改文件、命令执行和其他有副作用工具默认更倾向 ask 或 deny，具体行为由权限模式和规则共同决定。
- F26: LunaCode 能加载规则配置并在配置格式错误时给出可理解错误。某一层规则文件加载失败时，只忽略该失败层，其他已成功加载的规则层仍然生效；失败层不得被当作全放行。若本地级 `.lunacode/permissions.local.yaml` 损坏，始终允许无法自动追加规则，并提示用户先修复该文件。
- F27: LunaCode 提供可观察的权限决策结果。TUI 或 Agent 事件流能看到权限请求、权限允许、权限拒绝、命中规则、命中黑名单、沙箱拒绝或用户确认结果等状态。
- F28: 规则配置中的敏感路径、命令参数或环境变量值不应被完整泄露到普通回复中；展示给用户和模型的摘要应避免暴露 API Key、Token 或已知敏感值。敏感路径至少包括 `.lunacode/config.yaml` 主配置文件、`.lunacode/permissions.local.yaml` 本地权限规则文件和 `.lunacode/skills/` Skill 目录。
- F28a: 敏感路径默认触发 ask，而不是默认 deny。没有命中 deny 或 allow 规则时，访问 `.lunacode/config.yaml`、`.lunacode/permissions.local.yaml` 或 `.lunacode/skills/` 这类敏感路径必须交给用户确认；如果任意层 deny 命中则直接拒绝，如果 allow 命中则按规则放行。
- F29: 本阶段完成后，已有纯对话、工具调用、Agent Loop、Plan Mode、结构化 Prompt 和工具事件展示行为应保持可用。

## 非功能需求

- N1: 权限决策顺序必须稳定、可测试、可解释。相同工具调用、相同规则和相同权限模式应得到一致结果。
- N2: 权限系统默认失败策略应保守。黑名单解析失败、路径真实路径解析失败、规则文件格式错误或权限确认异常时，不得静默全放行。
- N3: 路径沙箱必须跨平台工作，至少兼容 Windows 路径分隔符、盘符大小写、相对路径、绝对路径和符号链接。
- N4: 权限规则匹配应足够简单，避免引入脚本执行、表达式求值或可执行配置，防止规则文件本身成为攻击面。
- N5: 权限系统应与具体工具实现解耦。新增工具时，只要提供工具名、参数摘要和风险属性，就能进入同一套权限流程。
- N6: 权限系统应与 Provider 和 TUI 解耦。模型供应商差异不影响权限判断；TUI 只消费权限事件和确认请求。
- N7: 人在回路确认应避免阻塞整个应用到不可恢复状态。用户拒绝、取消或输入无法识别时，应得到明确结果并继续正常交互。
- N8: 权限被拒绝后的工具结果应足够简洁，既能让模型调整策略，又不泄露敏感配置或过多本地环境细节。
- N9: 规则加载和权限决策应有自动化测试覆盖，包括黑名单、沙箱、规则优先级、权限模式、人在回路和拒绝后继续循环。
- N10: 本阶段不改变项目默认中文协作风格；用户可见权限说明、错误摘要和规则示例应使用中文。

## 不做的事

- 本阶段不实现细粒度网络访问策略，不按域名、IP、端口或协议控制外联访问；Linux Bash OS 级沙箱只提供默认断网和配置项整体开启网络。
- 本阶段不实现 CPU、内存、磁盘、子进程数量、命令运行时长之外的新资源配额系统。
- 本阶段不实现审计日志、合规报表、不可篡改日志或长期安全事件查询。
- 本阶段不实现插件市场、第三方权限策略插件或动态下载规则。
- 本阶段不实现复杂策略语言、条件表达式、脚本化规则或基于时间、用户身份、Git 分支的高级策略。
- 本阶段不实现图形化规则管理界面；规则以 YAML 文件和人在回路确认入口维护。
- 本阶段不要求支持多用户协作权限、远程工作区权限同步或企业集中策略下发。
- 本阶段不改变模型工具调用协议，也不新增业务工具；只把已有工具执行纳入新的权限决策流程。
- 本阶段不要求权限系统识别所有可能危险命令。黑名单只覆盖明确高危模式，其他未知风险通过规则、权限模式和人在回路处理。

## 验收标准

- AC1: 任意工具执行前都会经过统一权限决策，并能观察到 allow、deny 或 ask 结果。
- AC2: Bash 命令命中危险操作黑名单时，即使存在 allow 规则或处于 bypassPermissions 模式，工具调用仍被拒绝；文件工具不会因危险操作黑名单被拒绝，而是继续按路径沙箱、敏感路径、规则和权限模式判断。
- AC3: 黑名单拒绝结果包含明确错误类型和简短原因，并作为工具结果回灌给模型，而不是导致 Agent Loop 崩溃。
- AC3a: 在 Linux 上执行 Bash 命令时，LunaCode 会通过 bubblewrap 包裹命令并启用 seccomp；未配置 `sandbox.network_enabled: true` 时，命令运行在空网络命名空间中，无法访问网络。
- AC3b: 配置 `sandbox.network_enabled: true` 后，Linux Bash OS 级沙箱不再使用空网络命名空间，`npm install`、`pip install` 这类需要网络的命令可以在其它权限层允许后访问网络。
- AC3c: 即使开启 `sandbox.network_enabled: true`，Bash 黑名单、路径沙箱、deny 规则、权限模式和人在回路确认仍然生效。
- AC4: 文件工具传入工作区内普通相对路径时，路径沙箱允许继续后续权限判断。
- AC5: 文件工具传入 `..` 逃逸路径、允许根目录外绝对路径或指向允许根目录外的符号链接时，路径沙箱拒绝该工具调用；WriteFile 写入不存在的新文件时，必须先校验最近存在父目录的真实路径，父目录安全后才能继续判断目标文件。
- AC6: 命令工具传入允许根目录外 cwd 时，路径沙箱拒绝该命令执行；Bash 命令文本中出现可识别的允许根目录外绝对路径或 `..` 逃逸路径时，路径沙箱拒绝该命令执行。
- AC6a: 配置额外根目录后，文件工具和命令 cwd 传入额外根目录内路径时可以通过路径沙箱并继续后续权限判断；未配置的项目外路径仍被拒绝。
- AC6b: 额外根目录配置别名为 `cache` 且真实路径指向本机缓存目录时，该目录内路径在规则匹配中显示为 `/roots/cache/...`，并可被 `ReadFile(/roots/cache/**)` 这类规则匹配。
- AC7: 用户级规则项 `rule: "Bash(git *)"` 且 `effect: allow` 能按命令文本匹配 git 命令；规则项 `rule: "ReadFile(/project/src/**)"`、`rule: "WriteFile(/project/spec/**)"`、`rule: "EditFile(/project/src/**/*.java)"` 能按 `/project/...` 虚拟根路径匹配对应文件工具；规则项 `rule: "Glob(/project/src/**/*.java)"` 能按规范化后的 glob 入参或搜索范围匹配；规则项 `rule: "Grep(*password*)"` 且 `effect: deny` 能按搜索 pattern 拒绝敏感搜索。
- AC8: 任意层级的 deny 规则都能拒绝匹配操作；即使本地级存在 allow，也不能覆盖项目级或用户级中已经命中的 deny。
- AC9: 在没有任何 deny 命中的情况下，本地级 allow 规则能覆盖项目级或用户级默认规则，并且不会要求提交到版本控制。
- AC10: 同一层规则中，后定义的匹配规则优先级更高；多个 allow 同时匹配时采用靠后的 allow，多个 deny 同时匹配时采用靠后的 deny 作为拒绝原因，但任何 allow 都不能覆盖已经命中的 deny。
- AC11: default 模式下，未被具体规则允许的写文件、改文件和命令执行会触发 ask，而不是直接执行。
- AC12: 默认模式下，只读工具在路径沙箱通过后可以顺畅执行；写文件、改文件和命令执行仍需要规则允许或用户确认。
- AC13: bypassPermissions 模式下，大多数未命中操作可以自动放行；危险操作黑名单和路径沙箱仍然生效。
- AC13a: 用户输入 `/permissions` 能看到当前权限模式；输入 `/permissions acceptEdits` 后，后续文件编辑工具调用按 acceptEdits 模式决策；输入 `/permissions bypassPermissions` 时必须二次确认，取消确认后权限模式保持不变；当前权限模式能在状态栏、权限确认提示或 Agent 事件中被观察到。
- AC14: 规则没有明确命中时，TUI 显示权限确认请求，包含工具名、参数摘要、目标路径或命令摘要和可选动作。
- AC15: 用户选择本次允许后，当前工具调用执行成功，但同类下一次调用仍会重新请求确认。
- AC16: 用户选择始终允许后，LunaCode 将 allow 规则写入 `.lunacode/permissions.local.yaml`，同一项目中匹配的后续工具调用不再重复询问。
- AC17: 重启 LunaCode 后，本地级 `.lunacode/permissions.local.yaml` 中由始终允许生成的规则仍能被加载。
- AC17a: 用户对 `Bash(git status --short)` 选择始终允许后，自动写入的规则精确匹配 `Bash(git status --short)`，不会自动放宽成 `Bash(git status*)` 或 `Bash(git *)`。
- AC18: 用户拒绝权限请求后，工具不会执行，Agent Loop 继续运行，并把权限拒绝工具结果提供给模型。
- AC19: 模型在收到权限拒绝后可以改用更安全的工具或路径继续完成请求，而不是整轮请求直接中断。
- AC20: plan 模式下，写文件、改文件和命令执行的权限行为与 default 一致，会触发确认而不是自动放行。
- AC21: 某一层规则文件格式错误时，LunaCode 给出可理解错误，只忽略该失败层，其他规则层继续生效，不会把失败层当作全放行；本地级规则文件损坏时，始终允许不会静默写入失败文件。
- AC22: 权限事件流或 TUI 中能观察到权限允许、权限拒绝、权限请求、黑名单命中、沙箱拒绝和用户确认结果。
- AC23: 权限提示、错误摘要和工具结果不会泄露配置中的 API Key、Token 或其他已知敏感值；对 `.lunacode/config.yaml`、`.lunacode/permissions.local.yaml` 和 `.lunacode/skills/` 的访问会被视为敏感路径访问，未命中具体规则时默认触发 ask。
- AC24: 自动化测试覆盖黑名单硬拒绝、符号链接逃逸、不存在文件的父目录校验、Bash 可识别路径片段校验、额外根目录、deny 跨层不可翻转、allow 规则优先级、规则优先于权限模式、人在回路本次允许、始终允许、拒绝后继续 Agent Loop。
- AC25: 使用 checklist 中指定的端到端场景运行 LunaCode，输入真实请求“读取当前项目的 pom.xml，然后尝试执行一个未配置允许的写入操作”，可观察到读取正常执行，写入触发权限确认，用户拒绝后 Agent Loop 继续给出中文解释和替代方案。
