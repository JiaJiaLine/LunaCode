# LunaCode 工具系统 Spec

## 背景

LunaCode 第一阶段已经具备终端 TUI 对话、流式模型响应、Provider 抽象、对话历史管理和基础错误展示能力，但当前模型只能生成文本回复，不能读取项目、修改文件、执行命令或搜索代码。本阶段的目标是给 LunaCode 增加工具系统，让模型在一次用户请求中可以识别需要调用的工具，由 LunaCode 执行工具，再把结构化结果回灌给模型生成最终回复。

用户初稿中提到的 MewCode 在本文中按当前仓库项目名 LunaCode 处理。本阶段承接 `spec/01` 中明确未实现的 tool use 能力，但只完成一次工具调用和一次结果回灌，不实现自动循环式 Agent Loop。

## 目标

- LunaCode 提供一套统一工具抽象，每个工具都能描述自身名称、用途、参数结构和执行行为。
- LunaCode 内置六个核心工具：读取文件、写入文件、替换修改文件、按模式查找文件、搜索代码内容、执行命令。
- LunaCode 能集中登记工具，并把已登记工具转换为模型 API 可以识别的工具声明。
- 模型流式响应中出现工具调用时，LunaCode 能解析工具名和 JSON 参数，执行对应工具，并把结果作为工具结果消息加入对话历史。
- 工具执行失败时，LunaCode 不崩溃，而是把失败原因、错误类型和可重试提示包装成结构化结果返回给模型。
- 文件替换修改必须基于原文唯一匹配；匹配不到或匹配多次时，工具返回清晰错误，便于模型调整参数后重试。
- 命令执行必须有超时和错误处理，避免长时间卡住 TUI 或后台流式流程。
- 本阶段工具调用只执行一轮：模型请求工具，LunaCode 执行工具并回灌结果，再请求模型生成最终回复后停止。

## 功能需求

- F1: LunaCode 提供统一工具抽象，工具元信息包含工具名称、自然语言描述、参数 Schema 和执行入口。
- F2: LunaCode 提供读取文件工具。模型传入文件路径后，工具返回带行号的文件文本内容或可理解错误；工具支持 `offset` 和 `limit` 参数，用于指定从第几行开始读取、最多读取几行，使模型可以分段读取长文件。
- F3: LunaCode 提供写入文件工具。模型传入文件路径和完整内容后，工具先递归创建父目录并写入目标文件，父目录权限为 0755，文件权限为 0644，最后返回写入结果。
- F4: LunaCode 提供替换修改文件工具。模型传入文件路径、待匹配原文和替换文本后，工具只在原文唯一匹配时修改文件。
- F5: LunaCode 提供命令执行工具。模型传入命令后，工具执行并返回退出码、标准输出、标准错误和超时状态。
- F6: LunaCode 提供按模式查找文件工具。模型传入 glob 或等价文件匹配模式后，工具返回匹配文件列表，并支持 `**` 递归匹配。
- F7: LunaCode 提供代码内容搜索工具。模型传入搜索模式和可选路径范围后，工具返回匹配位置和文本摘要。
- F8: LunaCode 提供工具注册中心，集中登记内置工具，按名称查找工具，并输出模型 API 所需的工具声明列表。
- F9: LunaCode 调用模型时带上已登记工具声明，使模型可以选择是否请求工具调用。
- F10: LunaCode 能解析模型流式响应中的工具调用信息，包括工具名、调用 ID 和分片到达的 JSON 参数。
- F11: JSON 参数以流式碎片到达时，LunaCode 能拼接完整参数文本，并在工具调用结束时解析为结构化参数。
- F12: LunaCode 能根据工具名查找工具并执行；工具不存在、参数非法、执行失败、超时或权限受限时，均返回结构化工具结果。
- F13: LunaCode 把工具执行结果加入对话历史，使模型能基于工具结果生成面向用户的最终回复。
- F14: 工具结果消息必须能关联到对应工具调用 ID，保证模型可以识别结果属于哪一次调用。
- F15: 工具执行期间 TUI 应展示可观察状态，例如正在调用哪个工具、工具执行成功或失败。
- F16: 工具输出过长时，LunaCode 应返回受控长度的结果摘要，避免一次工具结果撑爆模型上下文或 TUI。
- F17: 本阶段完成一次工具调用结果回灌后，只再请求模型生成最终回复，不继续自动处理第二轮工具调用。

## 非功能需求

- N1: 工具执行必须有超时控制；超时后返回结构化错误，不阻塞 LunaCode 长时间不可用。
- N2: 工具执行错误必须可诊断，至少区分工具不存在、参数错误、文件不存在、匹配失败、匹配不唯一、命令超时和命令非零退出。
- N3: 文件类工具只能在 LunaCode 当前工作区或允许的路径范围内操作，避免模型随意读写无关系统路径。
- N4: 写文件和改文件工具在父目录创建、权限设置或内容写入失败时，不应留下半写入或不可解释的中间状态。
- N5: 命令执行工具返回信息不应泄露配置中的 API Key 或其他已知敏感配置值。
- N6: 工具注册和工具声明生成应与具体 Provider 解耦，避免每新增一个工具都要修改 TUI 或 Provider 外层编排逻辑。
- N7: OpenAI 和 Anthropic 的工具调用事件差异应被映射到 LunaCode 内部统一的工具调用表示。
- N8: 工具调用解析和执行应有自动化测试覆盖，包括流式参数拼接、成功执行、失败执行和结果回灌。
- N9: 本阶段应保持第一阶段纯对话能力可用；当模型没有请求工具时，LunaCode 仍按普通流式对话工作。

## 不做的事

- 本阶段不实现多轮自动 Agent Loop；工具结果回灌后若模型继续请求工具调用，LunaCode 不自动连环执行。
- 本阶段不实现并行多工具调用；即使模型返回多个工具调用，也只支持按本阶段约定处理单次工具调用或给出明确不支持结果。
- 本阶段不实现长期任务队列、后台任务恢复、跨进程工具执行状态持久化。
- 本阶段不实现复杂补丁格式、语义级代码修改或 AST 重写；文件修改仅支持原文唯一匹配替换。
- 本阶段不实现 shell 交互式会话、持续运行服务管理、终端复用或 tmux 自动化能力本身。
- 本阶段不新增插件市场或第三方工具动态加载；只做内置工具注册。
- 本阶段不实现精细权限弹窗或逐工具用户确认流程；权限边界先由工作区范围和工具执行限制保证。
- 本阶段不要求支持图片、二进制文件或富媒体工具结果；文件读取和搜索以文本内容为主。

## 验收标准

- AC1: LunaCode 启动后，模型 API 请求中包含六个核心工具的声明，声明包含名称、描述和参数 Schema。
- AC2: 模型请求读取文件时，LunaCode 执行读取文件工具，并把带行号的文件内容或文件错误作为工具结果回灌给模型。
- AC3: 模型请求写入文件时，LunaCode 递归创建父目录、写入目标文本文件，并返回成功状态、路径和写入字节数或字符数；父目录权限为 0755，文件权限为 0644。
- AC4: 模型请求替换修改文件且原文只出现一次时，LunaCode 修改文件并返回成功结果。
- AC5: 模型请求替换修改文件但原文不存在时，LunaCode 不修改文件，并返回“匹配不到”的结构化错误。
- AC6: 模型请求替换修改文件但原文出现多次时，LunaCode 不修改文件，并返回“匹配不唯一”的结构化错误。
- AC7: 模型请求执行命令时，LunaCode 返回退出码、标准输出、标准错误和是否超时。
- AC8: 命令执行超过配置或默认超时时间时，LunaCode 终止该次执行并返回超时错误，TUI 不长时间卡死。
- AC9: 模型请求按模式查找文件时，LunaCode 返回匹配文件列表，并能用 `**` 匹配任意层级子目录；没有匹配时返回空列表而不是崩溃。
- AC10: 模型请求搜索代码内容时，LunaCode 返回包含文件路径、匹配位置和匹配摘要的结果。
- AC11: 模型流式输出工具调用参数碎片时，LunaCode 能正确拼接 JSON 参数，并在工具调用结束后执行工具。
- AC12: 工具名不存在或参数不符合 Schema 时，LunaCode 返回结构化失败结果，并允许模型基于错误继续生成最终回复。
- AC13: 工具执行结果加入对话历史后，最终模型回复能引用工具结果回答用户，而不是只显示原始工具 JSON。
- AC14: 当模型没有请求工具调用时，LunaCode 保持第一阶段的普通流式对话行为。
- AC15: 当模型在工具结果回灌后的最终回复中再次请求工具调用时，LunaCode 不进入自动循环，并给出本阶段不支持连环工具调用的可理解状态或回复。
- AC16: 模型请求读取文件并传入 `offset` 和 `limit` 时，LunaCode 只返回指定行范围，并且每行结果都包含原始行号。
- AC17: 使用 checklist 中指定的端到端场景运行 LunaCode，输入真实请求“读取当前项目的 pom.xml 并总结依赖”，可观察到模型请求读文件工具、工具执行、结果回灌和最终中文回复。
