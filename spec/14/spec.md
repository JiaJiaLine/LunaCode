# Agent Team 与 Coordinator Mode Spec

## 背景

LunaCode 已经具备子 Agent、后台任务、定义式角色、Fork 子 Agent 和 Git Worktree 隔离能力。当前主 Agent 可以把单个子任务派给子 Agent，但缺少一个长期存在的团队协作层：多个队员不能围绕同一目标共享任务清单、直接互发消息、持续保留上下文，也没有一个让 Lead 专注调度和合并成果的纪律模式。

本阶段要把主 Agent 升级为 Team Lead。Lead 能创建长期团队，派生多个成员并行干活；成员之间通过共享任务列表和邮箱协作，不要求所有信息都经 Lead 中转。团队成员默认使用独立 worktree 工作，完成后由 Lead 基于 Git 分支合并成果。Coordinator Mode 作为独立可选能力，用 feature flag 和用户环境变量双重开启，开启后收窄 Lead 的工具集，让 Lead 专注拆解、派人、阅读结果、合成实施规格和合并代码。

## 目标

- 提供长期存在的团队对象，记录团队名、负责人、成员花名册、持久化位置和仓库身份。
- 支持 Lead 通过 `/team` 命令族创建、查看、切换、管理团队，并触发任务、消息、成员和合并相关操作。
- 为团队成员提供共享任务清单和点对点邮箱，让成员能直接协作、认领可执行任务、报告进展和请求审批。
- 让团队成员默认拥有独立 worktree，避免并行文件修改互相覆盖，并让 Lead 能基于成员分支合并成果。
- 支持成员自然停下后进入空闲状态，后续 Lead 发送消息时能从持久化上下文恢复成员继续工作。
- 提供 Coordinator Mode，在双重开关开启后剥夺 Lead 的写文件工具，只保留调度、阅读和必要的 shell/git 能力。
- 支持由 Lead 推理驱动的任务拆解、依赖建模、成员派生、结果综合和 Git 合并流程。

## 功能需求

- F1: 系统必须提供一个长期存在的团队对象，至少记录团队名称、负责人、成员列表、仓库身份、持久化目录和创建/更新时间。
- F2: 团队必须按仓库隔离持久化到用户目录下；同一用户在不同仓库中可以使用相同团队名而不冲突。
- F3: 团队持久化目录必须包含团队元数据、成员元数据、任务清单、成员邮箱和名称注册表。
- F4: 系统必须提供 `/team` 命令族作为用户可见入口，覆盖创建、删除、列出、使用、查看状态、管理成员、管理任务、发送消息和合并成员成果。
- F5: 团队成员元数据必须记录成员名、Agent ID、角色、运行后端、worktree 路径、worktree 分支、是否需要计划审批、当前状态、最近活动时间和恢复所需上下文位置。
- F6: 团队成员名称必须通过安全校验，并在同一团队内唯一；名称注册表必须能把成员名解析到成员元数据和邮箱文件。
- F7: 第一版成员运行后端必须支持同进程协程或后台任务方式运行完整成员循环。
- F8: 系统必须抽象成员运行后端选择，并能探测终端窗格后端是否可用；第一版请求终端后端但环境不可用或尚未实现时，必须返回明确错误，不得静默降级到同进程后端。
- F9: 成员默认必须使用独立 worktree 工作；团队成员元数据必须记录对应 worktree 路径和分支。
- F10: 团队成员 worktree 创建、保留、清理和删除保护必须复用上一阶段 worktree 隔离的安全规则。
- F11: Lead 创建成员时，如果指定 `agentType`，成员必须走定义式角色路径，从空白上下文启动，并使用对应角色定义。
- F12: Lead 创建成员时，如果未指定 `agentType` 且 `FORK_SUBAGENT` feature flag 已启用，成员必须走 Fork 路径，继承 Lead 的完整对话历史和需求背景。
- F13: Lead 创建成员时，如果未指定 `agentType` 且 `FORK_SUBAGENT` feature flag 未启用，系统必须使用默认 `general-purpose` agent，而不是触发 Fork。
- F14: Lead 必须由自身推理拆解用户目标，判断可并行任务和依赖关系，然后写入共享任务清单；系统不替 Lead 自动生成任务语义。
- F15: 共享任务清单必须支持创建、读取、列表和更新任务，任务至少包含 ID、标题、描述、状态、负责人、依赖、阻塞对象、创建者、更新时间和结果摘要。
- F16: 任务依赖必须支持系统级结构化表达；更新任务时可以通过 `addBlocks` 和 `addBlockedBy` 建立有向依赖关系。
- F17: 任务依赖也必须支持文本约定；Lead 可以在任务描述中写明非强制的顺序说明，由成员阅读后自主判断。
- F18: 成员通过任务列表必须能看到哪些任务被阻塞、哪些任务可执行、哪些任务已分配给自己、哪些未分配任务可以认领。
- F19: 任务分配必须支持 Lead 显式指定负责人；未分配且未被系统级依赖阻塞的任务可以由成员认领，认领必须避免并发覆盖。
- F20: 团队协作工具必须只暴露给 Team Lead 和注册团队成员；普通主入口对话和普通子 Agent 不得看到任务管理、团队管理和邮箱协作工具。
- F21: 同进程团队队员除自身干活工具外，必须额外获得固定协作工具集：`TaskCreate`、`TaskGet`、`TaskList`、`TaskUpdate` 和 `SendMessage`，用于查看/更新任务、认领任务、发送点对点消息、广播消息和报告完成状态。
- F22: 点对点消息必须走“Agent 名称注册表 + 邮箱文件”的两段式流程；`Agent` 工具的 `name` 参数非空时，系统必须注册 `name -> agentId` 映射，团队队员创建时必须自动完成注册，`SendMessage` 通过名称或 Agent ID 找到目标队员后再投递到目标邮箱。
- F23: 邮箱消息必须至少包含发件人、收件人、消息类型、正文、结构化载荷、时间戳、已读状态和摘要；落盘时系统必须自动补时间戳并默认未读。
- F24: 发送纯文本消息时，调用方必须提供 `summary` 字段；摘要必须是 5-10 个词，用作 UI 消息预览，便于 Lead 和队员快速扫描消息列表。
- F25: `SendMessage` 的 `to` 参数必须支持队友名称和 Agent ID 两种单目标寻址；名称通过 agent name registry 解析成 Agent ID，两者必须路由到同一个成员邮箱。
- F26: `SendMessage` 的 `to` 参数必须支持 `"*"` 广播，向当前团队内所有队友发送同一条消息。
- F27: 当目标 Agent 已停止但存在可恢复上下文时，发送消息必须触发从磁盘恢复该目标，而不是静默丢弃消息。
- F28: 邮箱文件写入必须使用锁文件保证并发安全；拿不到锁时必须重试，锁太旧时必须视为过期锁并允许恢复。
- F29: 消息系统必须支持单播、广播和结构化协议消息；结构化协议至少覆盖任务通知、计划审批请求、计划审批回复、完成报告、空闲通知、优雅退出请求和优雅退出回复。
- F30: `shutdown_request` 必须表示请求目标队员优雅退出；目标队员可以回复 `shutdown_response` 表示批准或拒绝，并附带原因。
- F31: `shutdown_response` 只能发送给 Lead；非 Lead 目标必须被拒绝。
- F32: `plan_approval_response` 必须包含批准或驳回结果和可选反馈，并且只有 Lead 可以发送。
- F33: 第一版独立终端窗格后端不要求可用，但如果后续后端声明需要唤醒目标窗格，消息发送流程必须为唤醒动作预留可观测结果。
- F34: 成员设置 `planModeRequired` 后，在执行任何修改操作前必须先通过邮箱向 Lead 发送执行计划。
- F35: Lead 必须通过结构化 `plan_approval_response` 回复审批结果；回复必须表达批准或驳回，并可附带反馈。
- F36: 计划审批通过后，Lead 当前权限模式必须传递给对应成员，成员再开始执行修改操作。
- F37: 计划审批被驳回后，成员不得继续执行修改操作，必须根据反馈调整计划或等待新指令。
- F38: 成员完成当前任务并自然停下后，系统必须把成员标记为空闲，并通知 Lead。
- F39: 空闲成员的上下文必须可从磁盘恢复；Lead 后续发送消息或分配任务时，系统必须恢复该成员继续工作，而不是从空白状态重新 spawn。
- F40: Coordinator Mode 必须独立于 Agent Team 存在，并由 `COORDINATOR_MODE` feature flag 和 `LUNACODE_COORDINATOR_MODE` 环境变量双重开启。
- F41: 只有当 feature flag 启用且环境变量为 truthy 时，Lead 才进入 Coordinator Mode；任一开关关闭都不得改变 Lead 的默认行为。
- F42: Coordinator Mode 开启后，Lead 的可用工具必须收窄为调度、团队管理、任务管理、消息、阅读类工具和 Bash；写文件、编辑文件等直接代码修改工具不得暴露给 Lead。
- F43: Coordinator Mode 必须保留 Bash，使 Lead 能执行 Git 合并、查看状态和必要的诊断命令。
- F44: Coordinator Mode 必须注入专用系统提示词，引导 Lead 以 spawn、收结果、synthesis、续写执行规格的阶段推进复杂任务。
- F45: Coordinator Mode 的提示必须要求 Lead 自己理解成员研究结果并写出具体实施规格，不得把理解和综合职责再次委托给成员。
- F46: Lead 必须能根据共享任务清单和成员完成状态，按依赖顺序或自身判断选择成员分支进行合并。
- F47: 合并由 Lead 通过 Bash 执行 Git 命令、通过 ReadFile 检查冲突文件，并由 Lead 自身判断合并顺序和冲突解决策略。
- F48: Lead 可以尝试解决机械性冲突；当冲突涉及逻辑矛盾或 Lead 没有把握时，必须回滚当前合并并向用户报告冲突上下文、成员分支和需要用户决策的问题。
- F49: 合并失败或用户需要介入时，成员 worktree 和分支必须保留，不能自动删除可能有价值的成果。
- F50: 团队删除必须默认保护成员成果；存在未合并分支、未提交修改、未读重要消息或运行中成员时，必须拒绝静默删除。

## 非功能需求

- N1: 团队、成员、任务、邮箱和 worktree 路径处理必须防路径遍历、防名称冲突，并始终限制在 LunaCode 管理目录内。
- N2: 团队协作状态必须可诊断；创建成员、发送消息、认领任务、审批计划、恢复成员和合并分支的失败原因必须可观测。
- N3: 邮箱和任务清单并发写入必须保护数据完整性；锁超时恢复不能导致消息丢失或任务状态倒退。
- N4: 成员后端选择不得静默降级；请求的隔离等级无法满足时必须明确失败。
- N5: 团队成员默认 worktree 隔离不能破坏现有普通子 Agent、Fork 子 Agent、Hook 子 Agent 和手动 worktree 行为。
- N6: Coordinator Mode 必须是显式 opt-in；未同时打开两个开关时，Lead 的工具集和行为提示保持现有语义。
- N7: Coordinator Mode 收窄工具时必须在声明层和执行层同时生效，避免模型看不到工具但仍能绕过执行层调用。
- N8: 合并流程必须优先保护工作成果；不能因为自动清理、团队删除或合并失败丢弃成员修改。
- N9: 第一版应复用现有 Agent、后台任务、工具执行、权限和 worktree 基础设施，不引入跨机器服务依赖。

## 不做的事情

- 不做跨机器或分布式团队。
- 不做成员之间实时流式通信；第一版通过邮箱文件和任务清单协作。
- 不做完整终端窗格后端实现；第一版只保留后端抽象、探测和明确失败行为。
- 不做复杂任务依赖求解器；系统级依赖只表达阻塞关系，不自动证明最优调度。
- 不做完全自动的语义冲突解决；Lead 只能在自己有把握时解决冲突，拿不准必须回滚并报告用户。
- 不让普通主入口对话和普通子 Agent 直接看到团队协作工具。
- 不自动删除带有未合并成果、未提交修改或未读重要消息的团队目录、成员 worktree 或成员分支。

## 验收标准

- AC1: 用户通过 `/team create refactor` 创建团队后，用户目录下按当前仓库身份和团队名生成持久化目录，并包含团队元数据。
- AC2: 两个不同仓库中创建同名团队时，持久化目录互不冲突，列出团队时只显示当前仓库对应团队。
- AC3: 用户通过 `/team member add alice --agentType reviewer` 添加成员后，团队元数据记录 alice 的角色、后端、状态和 worktree 信息，并且 `agentNameRegistry` 中存在 `alice -> agentId` 映射。
- AC4: 用户添加成员时省略 `agentType` 且 `FORK_SUBAGENT` 已启用，成员继承 Lead 对话背景；该 flag 未启用时，成员使用默认 `general-purpose` agent。
- AC5: 用户请求终端窗格后端但当前环境不可用或第一版未实现时，系统返回明确错误，并且不会悄悄改用同进程后端。
- AC6: 成员启动后拥有独立 worktree，成员文件操作发生在自己的 worktree 路径下，不影响 Lead 当前工作目录。
- AC7: Lead 创建任务 A、B、C、D 后，可通过 `addBlockedBy` 让 C 等待 A、D 等待 B；成员查看任务列表时能看到阻塞状态。
- AC8: Lead 在任务描述中写入“等 A 完成后再开始”时，该文本说明会原样保留在任务中，供成员阅读判断。
- AC9: Lead 可以把任务显式分配给 alice；alice 的任务列表能看到该任务归自己处理。
- AC10: 未分配且未阻塞的任务可以被一个成员认领；并发认领同一任务时只能有一个成员成功。
- AC11: 普通主入口对话和普通子 Agent 的工具声明中不包含团队任务、邮箱和团队管理工具。
- AC12: Team Lead 的工具声明中包含团队管理、任务和消息工具；注册团队成员的工具声明中额外包含且仅包含协作工具 `TaskCreate`、`TaskGet`、`TaskList`、`TaskUpdate` 和 `SendMessage`。
- AC13: alice 向 bob 发送纯文本消息但未提供 5-10 词 `summary` 时，工具调用被拒绝并提示摘要要求。
- AC14: alice 向 bob 发送带 `summary` 的纯文本消息后，系统通过 `agentNameRegistry` 找到 bob 的 Agent ID 和邮箱文件，追加一条默认未读且带时间戳的消息，UI 预览显示该摘要。
- AC15: alice 直接向 bob 的 Agent ID 发送消息时，消息路由到与 `to="bob"` 相同的邮箱，证明名称和 Agent ID 共用同一注册表路由。
- AC16: `to="*"` 广播消息会写入所有队友邮箱，发送方能看到每个目标的写入结果或失败原因。
- AC17: 目标成员已停止但存在持久化上下文时，发送消息会触发恢复该成员继续处理。
- AC18: 邮箱锁被短暂占用时，发送消息会重试；遇到过期锁时，系统能恢复并继续写入，且不会破坏既有消息。
- AC19: 队员收到 `shutdown_request` 后，可以回复 `shutdown_response` 表达批准或拒绝及原因。
- AC20: 非 Lead 目标收到 `shutdown_response` 或非 Lead 发送 `plan_approval_response` 时，系统拒绝该协议消息。
- AC21: `planModeRequired` 成员尝试修改文件前，会先向 Lead 发送计划审批请求，而不是直接执行修改工具。
- AC22: Lead 回复批准的 `plan_approval_response` 后，成员继承 Lead 当前权限模式并继续执行任务。
- AC23: Lead 回复驳回的 `plan_approval_response` 后，成员不执行修改操作，并能收到反馈文本。
- AC24: 成员完成任务并自然停下后，任务状态更新为完成，成员状态变为空闲，Lead 收到空闲或完成通知。
- AC25: Lead 稍后给空闲成员发送新消息时，系统从持久化上下文恢复成员继续处理，而不是创建一个全新空白成员。
- AC26: 只开启 `COORDINATOR_MODE` feature flag 或只设置 `LUNACODE_COORDINATOR_MODE` 环境变量时，Lead 不进入 Coordinator Mode。
- AC27: 两个 Coordinator Mode 开关都开启时，Lead 看不到写文件和编辑文件工具，但仍能使用 Agent、团队、任务、消息、ReadFile、Glob、Grep 和 Bash。
- AC28: Coordinator Mode 中，Lead 会收到专用提示，要求先收集成员结果，再由 Lead 自己综合成具体实施规格。
- AC29: Lead 可按成员分支执行 Git 合并；无冲突或 Lead 能解决机械性冲突时，合并提交成功并记录合并结果。
- AC30: 合并遇到 Lead 无法可靠判断的逻辑冲突时，系统回滚当前合并，保留成员 worktree 和分支，并向用户报告冲突文件、相关成员和待决问题。
- AC31: 删除团队时，如果存在运行中成员、未合并成员分支、未提交修改或未读重要消息，系统默认拒绝删除并说明阻塞原因。
- AC32: 真实对话端到端测试中，Lead 能创建团队、拆解带依赖的任务、派生两个成员并行执行、通过邮箱收结果，并在成员完成后合并可合并成果。
