# LunaCode Agent Loop Spec

## 背景

LunaCode 第一阶段已经具备终端 TUI 对话、流式模型响应、Provider 抽象和对话历史管理能力。第二阶段已经补齐工具系统，使模型可以请求读取文件、写入文件、替换修改文件、查找文件、搜索代码内容和执行命令，并让 LunaCode 执行一次工具调用后把结果回灌给模型生成最终回复。

当前限制是：模型只能“一步一停”。当一次工具结果不足以完成任务时，LunaCode 不会继续让模型根据新结果决定下一步行动，用户必须反复追问或催促。本阶段目标是给 LunaCode 增加 Agent Loop，让它能按 ReAct 模式自主循环：先让模型思考和决定是否调用工具，执行工具，观察结果，再继续调整下一步，直到任务完成或触发停止条件。

用户初稿中提到的 MewCode 在本文中按当前仓库项目名 LunaCode 处理。本阶段承接 `spec/02` 明确未实现的多轮自动 Agent Loop、多个工具调用处理和 Plan Mode 能力，但不扩展到权限系统、上下文压缩或交互式确认。

## 目标

- LunaCode 能在一次用户请求中自动进行多轮 ReAct 循环，而不是每次工具回灌后停止。
- Agent Loop 每轮都能调用模型、收集流式输出、执行工具、回灌工具结果，并根据模型下一轮输出继续或停止。
- Agent Loop 的每一轮都抽象为独立状态机，并把“是否继续循环”的判断集中到一个决策点，便于后续扩展新的停止条件和模式。
- Agent Loop 每轮调用 Claude 时都传入 System Prompt，System Prompt 至少包含角色设定、环境信息和模式特定指令。
- Agent Loop 提供四类明确停止条件：模型主动表示已完成、达到迭代上限、用户取消、异常状态连续出现，避免无限循环、错误风暴或用户无法中断。
- Agent 和 TUI 通过 `AgentEvent` 异步事件流解耦：输入是用户消息，输出是一串可被 UI 消费的事件，界面只根据事件类型更新显示，不直接控制循环内部细节。
- 流式响应收集同时满足两个目的：实时把文本推给界面，并保留完整响应供循环判断和后续历史写入。
- 当模型一次返回多个工具调用时，LunaCode 能按工具安全性分批执行：可并发的安全工具并发执行，有副作用或不适合并发的工具串行执行。
- LunaCode 支持 Plan Mode 两段式工作流：`/plan` 阶段通过强 Prompt 约束模型先澄清需求、再探索和规划，并把计划写入指定 plan file；用户确认后，`/do` 阶段再执行计划。

## 功能需求

- F1: LunaCode 提供 Agent Loop 入口。输入是一条用户消息，输出是一个 `AgentEvent` 异步流；用户提交普通任务后，系统进入循环式处理，直到模型不再请求工具、达到停止条件或请求被取消。
- F2: Agent Loop 每一轮都以当前对话历史、可用工具声明和 System Prompt 调用 Claude，并消费模型流式响应。
- F3: Agent Loop 的每一轮必须有独立状态机，覆盖本轮开始、模型流式输出、工具调用收集、工具执行、工具结果回灌和本轮完成等状态。
- F4: Agent Loop 必须把“是否继续循环”的判断逻辑集中到一个地方，统一处理模型完成、迭代上限、取消信号、异常状态和是否存在下一批工具调用。
- F5: System Prompt 至少包含角色设定，说明 LunaCode 是终端环境中的 AI 编程助手，擅长阅读代码、编写代码和调试问题，并会先思考再行动，每一步解释推理过程。
- F6: System Prompt 至少包含环境信息，包括当前工作目录、操作系统和当前时间。
- F7: System Prompt 必须支持模式特定指令；当处于 Plan Mode 时，应追加 Plan Mode 指令。
- F8: 当模型本轮主动表示任务已完成，或本轮没有请求工具调用并给出最终回复时，Agent Loop 将本轮 assistant 回复视为最终回复，完成请求并停止循环。
- F9: 当模型本轮请求工具调用时，Agent Loop 执行工具，把工具结果按对应工具调用 ID 加入对话历史，再进入下一轮模型调用。
- F10: Agent Loop 支持迭代上限。达到上限时停止继续调用模型和工具，并向用户显示可理解的停止说明。
- F11: Agent Loop 支持用户取消信号传播。用户在 UI 层触发取消后，取消信号传递到 Agent Loop；Loop 在下一轮循环开始前检测到取消信号，干净退出，并把状态更新为已取消。
- F12: Agent Loop 支持异常状态检测。当模型请求不存在、拼写错误或已禁用的工具时，LunaCode 先把该问题包装成结构化错误工具结果回灌给模型，让模型自行调整。
- F13: Agent Loop 支持连续未知工具保护。当模型连续多次请求不存在、拼写错误或已禁用的工具并达到配置或默认阈值时，循环提前终止，并提示模型可能已经偏离可用工具范围。
- F14: Agent Loop 支持流式错误停止。模型流读取、事件解析或 Provider 返回错误时，当前请求停止，并把错误摘要显示给用户。
- F15: Agent Loop 对外发布 `AgentEvent` 异步事件流，事件类型包含 `stream_text`、`tool_use`、`tool_result`、`turn_complete`、`loop_complete`、`usage` 和 `error`。
- F16: `stream_text` 事件表示模型正在输出的文字增量，必须携带一小段文本，供 UI 追加到输出区域。
- F17: `tool_use` 事件表示模型请求调用工具，必须携带工具名、工具输入参数和请求 ID，供 UI 展示正在执行的工具及目标，例如 `ReadFile /project/main.py`。
- F18: `tool_result` 事件表示工具执行完成，必须携带执行结果、是否出错和耗时，供 UI 折叠展示工具结果并显示类似 `50ms` 的耗时信息。
- F19: `turn_complete` 事件表示一轮 LLM 调用完成，必须携带当前轮次序号，供 UI 更新本轮状态。
- F20: `loop_complete` 事件表示整个 Agent Loop 结束，必须携带总轮次，供 UI 结束本次交互状态。
- F21: `usage` 事件表示 Token 用量更新，必须携带累计输入 token 数和累计输出 token 数，供状态栏实时更新消耗。
- F22: `error` 事件表示发生错误，必须携带错误信息，供 UI 显示可理解的错误摘要。
- F23: TUI 只从 `AgentEvent` 异步流中消费事件，并根据事件类型更新界面；TUI 不直接依赖 Provider 原始事件、工具执行细节或循环控制逻辑。
- F24: 流式收集器在消费模型响应时走双路：收到文本增量时立即发布 `stream_text` 事件给界面，同时追加到本轮完整响应缓冲。
- F25: 流式收集器能收集本轮所有工具调用，包括工具调用 ID、工具名和完整 JSON 参数，并在本轮模型响应结束后交给 Agent Loop 判断。
- F26: 一次模型响应中包含多个工具调用时，Agent Loop 按工具安全性分批执行。只读且并发安全的工具可以同批并发执行；有副作用、破坏性或声明不适合并发的工具必须串行执行。
- F27: 多个工具调用执行完成后，Agent Loop 将每个工具结果按原工具调用 ID 回灌给模型，并保留成功、失败、超时和未知工具等结构化信息。
- F28: 工具执行期间，Agent 事件流能通过 `tool_use` 和 `tool_result` 展示每个工具的名称、输入参数、开始、成功、失败、耗时或错误摘要，使界面能观察当前进度。
- F29: Token 用量在循环中持续累计或更新。每轮模型返回用量时，Agent 事件流发布携带累计值的 `usage` 事件；无法获得精确用量时显示未知或部分统计。
- F30: Plan Mode 支持 `/plan` 指令。进入 `/plan` 后，系统向模型注入强指令，明确告知当前是规划模式，不能执行修改操作，不能编辑非 plan 文件，不能提交代码，不能修改配置。
- F31: Plan Mode 下，模型应按固定流程工作：先在需求不清楚时使用 AskUserQuestion 一步步澄清需求，再使用 ReadFile、Grep、Glob 和只读 Bash 命令探索代码，分析用户需求，设计实现方案，把计划写入指定 plan file，然后等待用户确认。
- F32: Plan Mode 下必须提供 AskUserQuestion 工具。该工具用于需求澄清，一次只提出一个聚焦问题，并把用户回答作为工具结果回灌给模型。
- F33: AskUserQuestion 只用于 Plan Mode 的需求澄清，不用于权限审批、执行确认或替代 `/do`。
- F34: Plan Mode 下的权限矩阵与 Default 模式保持一致：读操作允许，写操作需要确认，命令执行需要确认。
- F35: Plan Mode 唯一特殊权限处理是指定 plan file 自动放行。模型写入 plan file 不需要用户确认；模型尝试写入其他文件时，仍按 Default 模式触发权限确认。
- F36: `/plan` 生成的 plan file 路径和计划内容应保留在对话历史或 Agent 状态中，使 `/do` 能基于该计划继续执行。
- F37: Plan Mode 支持 `/do` 指令。用户确认执行后，LunaCode 切回普通执行语境，允许模型基于已有计划和当前权限策略进行实际修改或命令执行。
- F38: 当模型没有请求工具调用时，LunaCode 仍保持普通流式对话体验，不额外制造空工具步骤。

## 非功能需求

- N1: Agent Loop 必须有兜底安全网，默认迭代上限应足够防止无限循环，并能通过配置调整。
- N2: Agent Loop 运行期间 TUI 应保持可刷新，文本流、工具状态和取消状态都不应因为长时间工具执行或网络读取而卡住界面。
- N3: `AgentEvent` 事件模型应与 Provider、具体工具实现和 TUI 渲染解耦，新增事件消费者不需要修改核心循环行为。
- N4: 流式收集器应保证事件顺序对用户可理解：文本增量、工具调用、工具结果和完成状态不能出现明显倒置。
- N5: 多工具并发执行不得破坏工具已有安全边界；有副作用工具默认保守串行执行。
- N6: 工具批次执行失败时，不应让整个进程崩溃；失败应以结构化工具结果回灌给模型，除非触发明确停止条件。
- N7: 用户取消通过可传递的取消信号表达；Loop 不要求强行杀死当前已经开始的底层调用，但必须在下一轮循环开始前检查信号，并保证不会继续发起新的模型调用或工具批次。
- N8: 错误、停止说明和工具结果摘要不应泄露配置中的 API Key 或其他已知敏感值。
- N9: 本阶段应保持第一、二阶段能力可用：纯对话、单工具调用、工具结果回灌和基础 TUI 显示不得回退。
- N10: Plan Mode 依赖 Prompt 约束模型行为，但不能绕过已有权限边界；当模型不遵守规划指令并尝试修改非 plan 文件时，系统必须继续按默认权限流程处理。
- N11: 自动化测试应覆盖循环停止条件、每轮状态机、集中继续决策、System Prompt 构建与传递、事件流、流式收集、多工具分批、Plan Mode Prompt 注入、AskUserQuestion 需求澄清、plan file 自动放行和取消行为。

## 不做的事

- 本阶段不新建一套权限系统、权限弹窗、逐工具用户确认或策略审批；Plan Mode 只复用 Default 模式已有权限矩阵，并额外自动放行指定 plan file。
- 本阶段不实现上下文压缩、长对话摘要、向量检索或项目索引。
- 本阶段不实现复杂的用户交互式确认或权限审批流程；Plan Mode 只允许通过 AskUserQuestion 做需求澄清，`/plan` 到 `/do` 的两段式切换只要求用户确认是否进入执行，不做逐项计划审批。
- 本阶段不实现长期后台任务、任务恢复、跨进程 Agent 状态持久化或任务队列。
- 本阶段不新增第三方插件系统或动态工具加载；仍使用已有内置工具和注册机制。
- 本阶段不实现复杂权限沙箱；工具可访问范围沿用第二阶段已有工作区边界和工具限制。
- 本阶段不要求模型的“思考过程”可见；界面只展示模型正常输出、工具调用状态、工具结果摘要和进度事件。
- 本阶段不实现上下文过长时的自动裁剪策略；若达到模型上下文限制，按流式错误或 Provider 错误处理。
- 本阶段不要求支持 shell 交互式会话、持续运行服务管理或 tmux 自动化能力本身。

## 验收标准

- AC1: 用户提交一个需要多步完成的任务后，LunaCode 能自动进行多轮“模型输出 -> 工具执行 -> 工具结果回灌 -> 继续模型输出”，不需要用户每轮手动催促。
- AC2: 每一轮 Agent Loop 都通过本轮状态机推进，并能观察到本轮开始、模型流式输出、工具执行或本轮完成等状态变化。
- AC3: 模型完成、迭代上限、取消信号、异常状态和下一批工具调用都由同一个继续决策逻辑判断，而不是分散在多个调用点各自决定。
- AC4: 每轮调用 Claude 时，请求都包含 System Prompt。
- AC5: System Prompt 包含 LunaCode 的角色设定、当前工作目录、操作系统和当前时间。
- AC6: Plan Mode 下的 System Prompt 额外包含 Plan Mode 指令；Default 模式下不包含 Plan Mode 指令。
- AC7: 模型某一轮主动表示任务已完成，或没有请求工具调用并给出最终回复时，LunaCode 正常结束 Agent Loop，并将该轮 assistant 文本作为最终回复显示。
- AC8: 达到迭代上限时，LunaCode 停止继续调用模型和工具，并在对话区或状态栏显示达到迭代上限的说明。
- AC9: 用户在 UI 层触发取消后，取消信号传递到 Agent Loop；Loop 在下一轮循环开始前检测到取消信号，停止后续模型调用和工具批次，并显示已取消状态。
- AC10: 模型首次或少量请求不存在、拼写错误或已禁用的工具时，LunaCode 返回结构化错误工具结果，让模型有机会自行调整。
- AC11: 模型连续请求不存在、拼写错误或已禁用的工具达到阈值时，LunaCode 停止循环，并显示异常状态终止说明。
- AC12: Provider 流式响应出错或事件解析失败时，当前 assistant 消息进入错误状态，Agent Loop 停止，并显示可理解错误摘要。
- AC13: TUI 能通过 `stream_text` 事件实时显示文本增量，不需要等待整轮模型响应结束。
- AC14: 同一轮模型响应结束后，LunaCode 能拿到完整 assistant 文本和完整工具调用列表，用于判断是否继续循环。
- AC15: Agent 事件流能发布 `tool_use` 事件，事件中包含工具名、输入参数和请求 ID，TUI 能显示类似“正在执行 ReadFile /project/main.py”的提示。
- AC16: Agent 事件流能发布 `tool_result` 事件，事件中包含执行结果、是否出错和耗时，TUI 能折叠展示工具结果并显示类似 `50ms` 的耗时。
- AC17: Agent 事件流能发布 `usage` 事件，事件中包含累计输入 token 数和累计输出 token 数；当 Provider 没有提供完整用量时，状态栏显示未知或部分统计而不是报错。
- AC18: 每轮模型调用结束时，Agent 事件流发布 `turn_complete` 事件并携带当前轮次序号；整个循环结束时发布 `loop_complete` 事件并携带总轮次。
- AC19: 模型一次返回多个只读且并发安全的工具调用时，LunaCode 能在同一批次并发执行，并把每个结果按对应工具调用 ID 回灌。
- AC20: 模型一次返回包含写文件、改文件或其他有副作用工具的多个工具调用时，LunaCode 按安全顺序串行执行相关工具，不并发执行有副作用操作。
- AC21: 多工具批次中某个工具失败时，LunaCode 把该失败作为结构化工具结果回灌，其他已完成工具结果不丢失。
- AC22: 输入 `/plan` 后，发送给模型的系统指令包含 Plan Mode 强约束，明确要求模型先澄清需求，再探索代码和生成计划，并禁止修改非 plan 文件、提交代码或修改配置。
- AC23: Plan Mode 下，模型可以调用 AskUserQuestion 工具提出一个需求澄清问题，UI 显示该问题并收集用户回答，用户回答作为工具结果回灌给模型。
- AC24: Plan Mode 下，模型可以连续多轮使用 AskUserQuestion 逐步澄清需求；每次只提出一个聚焦问题，澄清后再生成 plan file。
- AC25: AskUserQuestion 不触发权限审批，也不能作为 `/do` 执行确认；它的工具结果只表示用户对需求问题的回答。
- AC26: Plan Mode 下，模型写入指定 plan file 时不触发用户确认，计划文件成功生成并可被后续 `/do` 使用。
- AC27: Plan Mode 下，模型尝试写入非 plan 文件时，LunaCode 按 Default 模式触发写操作确认，而不是静默执行或自动拒绝。
- AC28: Plan Mode 下，模型请求执行 Bash 命令时，LunaCode 按 Default 模式触发命令确认；用户拒绝后命令不执行，用户批准后按正常工具流程执行。
- AC29: `/plan` 生成的计划显示在对话中或可从指定 plan file 查看，并保留到后续对话历史或 Agent 状态。
- AC30: 输入 `/do` 后，LunaCode 切回普通执行语境，能基于已有计划执行需要写文件、改文件或命令执行的任务。
- AC31: 普通纯文本问题不触发工具调用时，LunaCode 保持第一阶段的流式回复体验。
- AC32: 第二阶段“读取当前项目的 pom.xml 并总结依赖”的端到端场景仍可运行，并能在 Agent Loop 中完成工具调用和最终中文回复。
- AC33: 使用 checklist 中指定的端到端场景运行 LunaCode，输入真实请求“检查当前项目里和工具执行有关的测试，并总结哪些测试覆盖了失败路径”，可观察到 LunaCode 自动多轮读取、搜索、汇总并给出最终中文回复。
